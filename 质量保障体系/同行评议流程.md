# 同行评议流程

## 1. 概述

同行评议是确保内容质量的重要机制，通过领域内的同行专家对文档进行评审，保证内容的技术正确性、创新性和实用性。

---

## 2. 评议类型

### 2.1 内容评议

**评议重点**：

- 技术内容正确性
- 概念表述清晰度
- 逻辑结构合理性
- 知识覆盖完整性

**适用对象**：

- 核心算法文档
- 理论定义文档
- 技术规范文档

### 2.2 形式评议

**评议重点**：

- 文档格式规范性
- 语言表达质量
- 图表使用恰当性
- 排版美观度

**适用对象**：

- 所有类型文档
- 特别适用于教程和指南

### 2.3 技术评议

**评议重点**：

- 代码/伪代码正确性
- 算法实现可行性
- 性能分析准确性
- 工程实践适用性

**适用对象**：

- 算法实现文档
- 性能分析文档
- 工程实践指南

---

## 3. 评议流程

### 3.1 流程概览

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 提交申请 │ → │ 分配评议人│ → │ 评议人评审│ → │ 作者回应  │ → │ 最终决议  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                      ↑              ↓
                                                      └──────────────┘
                                                           修改
```

### 3.2 详细流程

#### 步骤1：提交评议申请

**申请条件**：

- 文档已完成初稿
- 作者已完成自查
- 符合内容质量标准

**申请材料**：

```markdown
# 同行评议申请

## 基本信息
- **文档标题**：
- **文档类型**：算法/理论/规范/教程
- **作者**：
- **提交日期**：

## 文档摘要
[200字以内概述文档内容]

## 评议重点
[希望评议人关注的重点问题]

## 推荐评议人
[可选：推荐2-3位合适的评议人]
```

**提交方式**：

- 通过项目管理系统提交
- 或发送邮件至评议委员会

#### 步骤2：分配评议人

**分配原则**：

- 每份文档分配2-3名评议人
- 根据文档领域匹配专业背景
- 避免利益冲突

**评议人职责**：

- 独立进行评审
- 在规定时间内提交评议意见
- 保持评审的客观性和建设性

**时间要求**：

| 文档类型 | 评议周期 |
|---------|---------|
| 简短文档（<10页） | 1周 |
| 中等文档（10-30页） | 2周 |
| 长文档（>30页） | 3周 |

#### 步骤3：评议人评审

**评审维度**：

| 维度 | 权重 | 说明 |
|-----|------|------|
| 技术正确性 | 30% | 技术内容准确无误 |
| 内容创新性 | 20% | 有新颖见解或独特视角 |
| 表达清晰度 | 25% | 内容易于理解 |
| 实用价值 | 15% | 对读者有实际帮助 |
| 规范符合度 | 10% | 符合项目规范要求 |

**评审等级**：

| 等级 | 分数 | 说明 |
|-----|------|------|
| 强接受 | 9-10 | 质量优秀，可直接接受 |
| 接受 | 7-8 | 质量良好，小修改后接受 |
| 边界 | 5-6 | 有争议，需讨论 |
| 拒绝 | 3-4 | 需要重大修改 |
| 强拒绝 | 1-2 | 不适合本项目 |

#### 步骤4：作者回应

**回应流程**：

1. 收到评议意见后，作者逐条回应
2. 说明接受或拒绝每条建议的理由
3. 根据建议修改文档
4. 提交修改后的文档和回应表

**回应模板**：

```markdown
# 评议回应表

## 基本信息
- **文档标题**：
- **评议轮次**：第X轮
- **回应日期**：

## 逐条回应

### 评议人1：XXX

#### 意见1
**评议意见**：[原文]
**作者回应**：接受/部分接受/不接受
**修改说明**：已修改/未修改，原因...
**修改位置**：第X节第X段

#### 意见2
...

## 总结修改
1.
2.
3.
```

#### 步骤5：最终决议

**决议类型**：

| 决议 | 条件 | 后续操作 |
|-----|------|---------|
| **接受** | 平均分≥7，无重大反对意见 | 合并到主分支 |
| **修改后接受** | 平均分5-7 | 修改后维护者确认 |
| **修改后重审** | 平均分<5或有重大分歧 | 重大修改后重新评审 |
| **拒绝** | 平均分<3或存在根本性问题 | 文档退回 |

**决议流程**：

1. 维护者汇总所有评议意见
2. 计算综合评分
3. 组织评议人讨论（如有分歧）
4. 做出最终决议
5. 通知作者并说明理由

## 4. 评议标准

### 4.1 技术正确性（30%）

**评估要点**：

- [ ] 数学公式推导正确
- [ ] 算法描述逻辑正确
- [ ] 定理证明完整正确
- [ ] 引用来源准确可靠
- [ ] 技术细节无误

**评分指南**：

- 10分：技术内容完全正确，无可挑剔
- 8分：技术正确，有小问题
- 6分：基本正确，有需要澄清的地方
- 4分：有技术错误
- 2分：有重大技术错误

### 4.2 内容创新性（20%）

**评估要点**：

- [ ] 有独特见解
- [ ] 内容组织新颖
- [ ] 示例选取恰当
- [ ] 覆盖最新进展
- [ ] 有独到分析

**评分指南**：

- 10分：高度创新，填补空白
- 8分：有一定创新
- 6分：常规但完整
- 4分：缺乏新意
- 2分：陈词滥调

### 4.3 表达清晰度（25%）

**评估要点**：

- [ ] 结构清晰合理
- [ ] 语言流畅易懂
- [ ] 概念解释清楚
- [ ] 示例说明充分
- [ ] 图文配合得当

**评分指南**：

- 10分：表达清晰，易于理解
- 8分：基本清晰，个别地方需改进
- 6分：需要努力才能理解
- 4分：表达混乱
- 2分：难以理解

### 4.4 实用价值（15%）

**评估要点**：

- [ ] 对目标读者有帮助
- [ ] 可操作性强
- [ ] 能解决实际问题
- [ ] 有参考价值
- [ ] 内容完整可用

**评分指南**：

- 10分：极具实用价值
- 8分：有实用价值
- 6分：有一定参考价值
- 4分：实用价值有限
- 2分：几乎无实用价值

### 4.5 规范符合度（10%）

**评估要点**：

- [ ] 符合项目写作规范
- [ ] 格式统一规范
- [ ] 引用格式正确
- [ ] 术语使用一致
- [ ] 文档结构完整

**评分指南**：

- 10分：完全符合规范
- 8分：基本符合，小地方需调整
- 6分：有些地方不符合规范
- 4分：较多不符合规范
- 2分：严重不符合规范

## 6. 评议人指南

### 6.1 评议人资格

- 在相关领域有专业知识
- 了解项目背景和目标
- 有时间和精力完成评审
- 能保持客观公正

### 6.2 评议原则

1. **建设性**：批评要建设性，提出改进建议
2. **具体性**：意见要具体，指出位置和问题
3. **尊重性**：尊重作者劳动，语气礼貌
4. **保密性**：评审期间不泄露文档内容
5. **及时性**：在规定时间内完成评审

### 6.3 评议清单

**内容评审**：

- [ ] 已阅读全文
- [ ] 检查了技术正确性
- [ ] 评估了创新性
- [ ] 检查了引用准确性
- [ ] 评估了实用价值

**形式评审**：

- [ ] 检查了文档结构
- [ ] 检查了语言表达
- [ ] 检查了格式规范
- [ ] 检查了图表质量
- [ ] 检查了术语一致性

---

## 7. 特殊情况处理

### 7.1 评议分歧

当评议人之间存在重大分歧时：

1. 维护者组织评议人讨论
2. 必要时引入第三位评议人
3. 以多数意见为准
4. 维护者有最终裁决权

### 7.2 评议延期

评议人因故无法按时完成：

1. 提前通知维护者
2. 申请延期（最多延期1周）
3. 如无法完成，由维护者重新分配

### 7.3 利益冲突

评议人发现与作者存在利益冲突：

1. 立即声明利益冲突
2. 如冲突严重，退出评审
3. 由维护者重新分配评议人

---

## 8. 评议质量保障

### 8.1 评议人培训

- 新评议人需阅读本流程文档
- 参与示例评议培训
- 了解项目标准和期望

### 8.2 评议质量跟踪

- 跟踪评议完成时间
- 评估评议意见质量
- 收集作者对评议的反馈
- 持续改进评议流程

### 8.3 评议人激励

- 公开致谢评议贡献
- 优秀评议人认证
- 参与项目决策的机会
