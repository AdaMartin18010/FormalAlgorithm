# 季度审校流程 (Quarterly Review Process)

> **关联制度**：[同行评议流程](../质量保障体系/同行评议流程.md)
> **关联文档**：[内容所有权分配表](./内容所有权分配表_2026-04-29.md)、[模块状态仪表板](./模块状态仪表板.md)

## 1. 概述

### 1.1 目的

建立周期化的内容审校机制，确保每个模块的文档在准确性、完整性、引用和格式等方面持续符合项目质量标准。

### 1.2 适用范围

- 所有已分配 Module Owner 的编号模块和支持性核心目录
- 处于"活跃开发模式"和"维护模式"的模块
- 每季度第一个月集中执行

### 1.3 与同行评议流程的关系

| 维度 | 季度审校 | 同行评议 |
|:---|:---|:---|
| **触发时机** | 固定周期（每季度） | 文档发布前 / 重大修订时 |
| **执行人** | Module Owner | 独立评议人（双盲） |
| **关注点** | 模块整体健康度 | 单篇文档学术质量 |
| **产出物** | 审校报告 | 评议表格 + 修改响应表 |

季度审校发现的重要问题可触发同行评议流程，详见 [同行评议流程](../质量保障体系/同行评议流程.md) 第 2.1 节"提交阶段"。

## 2. 审校时间表

| 季度 | 审校月份 | 启动日期 | 截止日期 | 报告归档 |
|:---|:---|:---|:---|:---|
| Q2 2026 | 4 月 | 2026-04-01 | 2026-04-30 | 2026-05-05 |
| Q3 2026 | 7 月 | 2026-07-01 | 2026-07-31 | 2026-08-05 |
| Q4 2026 | 10 月 | 2026-10-01 | 2026-10-31 | 2026-11-05 |
| Q1 2027 | 1 月 | 2027-01-01 | 2027-01-31 | 2027-02-05 |

**说明**：

- 启动前 3 天，项目核心组发送审校提醒
- 截止后 5 天内，Owner 提交审校报告
- 超期未完成的模块，状态自动标记为"审校滞后"

## 3. 审校范围

每个 Module Owner 负责审校自己模块下的**所有文档**，包括：

- [ ] 模块根目录下的 Markdown 文件
- [ ] 所有子目录中的 Markdown 文件
- [ ] 附带的代码示例、配置文件、数据文件（如适用）

## 4. 审校检查点

### 4.1 准确性 (Accuracy)

- [ ] 技术概念、定理、公式无错误
- [ ] 算法描述与代码实现一致
- [ ] 复杂度分析正确
- [ ] 数学推导可复现

### 4.2 完整性 (Completeness)

- [ ] 文档结构符合 [模板规范](../项目维护/模板使用指南.md)
- [ ] 必要章节无缺失（定义、分析、证明、示例）
- [ ] 边界情况和异常处理有说明
- [ ] 与其他模块的交叉引用已建立

### 4.3 引用完整性 (Citation Integrity)

- [ ] 关键定义和定理标注出处
- [ ] 引用格式符合 ACM 规范
- [ ] 外部链接有效（抽查）
- [ ] 参考文献列表完整

### 4.4 链接有效性 (Link Validity)

- [ ] 模块内交叉引用可达
- [ ] 指向其他模块的链接有效
- [ ] 图片、附件路径正确
- [ ] 锚点引用准确

### 4.5 Frontmatter 完整性 (Frontmatter Completeness)

- [ ] `title` 字段存在且非空
- [ ] `version` 字段存在
- [ ] `last_updated` 字段存在且日期合理
- [ ] `owner` 字段存在且与所有权表一致
- [ ] `diataxis` 字段存在且取值有效（如适用）

## 5. 审校产出

### 5.1 审校报告模板

Owner 需填写以下报告并归档至 `docs/项目维护/审校报告/`：

```markdown
---
title: Q{X} 审校报告 - {模块名}
review_period: {起始日期} - {结束日期}
owner: {Module Owner}
module: {模块路径}
---

# Q{X} 审校报告 - {模块名}

## 审校统计

| 指标 | 数值 |
|------|------|
| 审校文档总数 | __ |
| 发现问题总数 | __ |
| 严重问题数 | __ |
| 一般问题数 | __ |
| 已修复数 | __ |
| 待修复数 | __ |

## 问题清单

| # | 文档路径 | 问题类型 | 严重程度 | 问题描述 | 修复计划 | 状态 |
|---|---------|---------|---------|---------|---------|------|
| 1 | | 准确性/完整性/引用/链接/Frontmatter | P0/P1/P2 | | | 待修复/已修复/延期 |

## 改进建议

1.
2.

## 下季度重点

1.
2.
```

### 5.2 问题分级

| 级别 | 定义 | 处理时限 |
|:---|:---|:---|
| P0 - 严重 | 技术错误、证明缺陷、安全漏洞 | 7 天内修复 |
| P1 - 一般 | 链接失效、格式错误、引用缺失 | 30 天内修复 |
| P2 - 轻微 | 排版、措辞、示例优化 | 下次审校前修复 |

## 6. 审校后流程

### 6.1 问题修复

1. Owner 根据审校报告制定修复计划
2. P0 问题需在截止日期后 7 天内修复
3. 修复完成后更新 `last_updated` 字段
4. 重大修复需记录在审校报告"修复确认"栏

### 6.2 状态更新

- 审校完成后，更新 [模块状态仪表板](./模块状态仪表板.md)
- 更新"最后审校日期"、"待修复 Issue 数"、"状态徽章"等字段
- 项目核心组汇总各模块报告，形成季度质量摘要

### 6.3 触发同行评议

若审校中发现以下情况，应启动 [同行评议流程](../质量保障体系/同行评议流程.md)：

- 涉及核心定理或算法正确性的重大修订
- 新模块首次发布前的质量把关
- 作者与 Owner 对修改方案存在分歧
