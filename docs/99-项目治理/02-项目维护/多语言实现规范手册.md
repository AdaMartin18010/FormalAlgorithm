# 多语言实现规范手册

> 版本：v1.0
> 适用范围：`examples/` 目录下的所有编程语言实现
> 目标：建立统一、可维护、可横向对比的多语言算法代码库

---

## 1. 文件组织规范（Rosetta Code 模式）

### 1.1 目录结构

每个算法应拥有独立的目录，目录内部按语言划分子目录。这是 [Rosetta Code](https://rosettacode.org/) 风格的组织方式，便于跨语言横向对比：

```
examples/
├── README.md                          # 总入口
├── CROSS_LANGUAGE_INDEX.md            # 跨语言对照索引
├── algorithms-rust/                   # Rust 实现库
│   ├── Cargo.toml
│   ├── src/
│   │   ├── lib.rs
│   │   ├── classic/                   # 经典算法（非 LeetCode）
│   │   │   ├── mod.rs
│   │   │   ├── merge_sort.rs
│   │   │   ├── dijkstra.rs
│   │   │   └── ...
│   │   └── leetcode/                  # LeetCode 题目实现
│   │       ├── mod.rs
│   │       ├── lc0001_two_sum.rs
│   │       └── ...
│   └── benches/
├── algorithms-python/                 # Python 实现库
│   ├── src/leetcode/
│   ├── src/sorting/
│   └── tests/
├── algorithms-go/                     # Go 实现库
│   ├── leetcode/
│   └── *_test.go
├── algorithms-java/                   # Java 实现库
├── algorithms-cpp/                    # C++ 实现库
├── algorithms-ts/                     # TypeScript 实现库
├── algorithms-js/                     # JavaScript 实现库
├── algorithms-c/                      # C 实现库
└── lean_proofs/                       # Lean 形式化证明
```

### 1.2 新增算法流程

1. 在 `CROSS_LANGUAGE_INDEX.md` 中登记题目/算法
2. 在对应语言的目录中创建实现文件
3. 确保所有语言版本的**核心思路一致**，仅语法/惯用法不同
4. 更新本手册（如引入新的范式或规范）

## 2. 命名规范

### 2.1 文件名

| 类型 | 格式 | 示例 |
|------|------|------|
| LeetCode 题目 | `lc{4位题号}_{snake_case简称}.{ext}` | `lc0001_two_sum.rs` |
| 经典算法 | `{snake_case算法名}.{ext}` | `merge_sort.rs` |
| 测试文件 | `test_{模块名}.{ext}` 或 `{模块名}_test.{ext}` | `test_merge_sort.py`, `merge_sort_test.go` |

### 2.2 函数/方法名

- **统一使用 `snake_case`**
- 即使在 Java / C++ 等惯用 `camelCase` 的语言中，也**建议**使用 `snake_case`，或在文件头部添加如下注释说明：

```java
// Naming Convention: 本文件遵循项目统一规范，使用 snake_case 命名函数，
// 以与 Rust/Python/Go 等语言保持一致，便于跨语言对比。
```

### 2.3 注释标记规范

```rust
// LeetCode 11. Container With Most Water
// Time Complexity: O(n), Space Complexity: O(1)
// Paradigm: Two Pointers / Greedy
```

必须包含的三行注释：

1. **题号标注**：`// LeetCode {题号}. {英文题目名}`
2. **复杂度标注**：`// Time Complexity: {复杂度}, Space Complexity: {复杂度}`
3. **范式标注**：`// Paradigm: {算法范式}`（如 Hash Table, Two Pointers, Dynamic Programming, Backtracking 等）

## 3. 代码风格规范

### 3.1 函数文档注释（必含）

每个公共函数/方法必须包含完整的文档注释，至少覆盖以下四部分：

```rust
/// {一句话功能描述}
///
/// # 形式化规约
///
/// - **前置条件 (Pre)**：输入应满足的条件
/// - **后置条件 (Post)**：输出应保证的性质
///
/// # 核心思路
///
/// {算法思路的简要描述，2-3 句话}
///
/// # 循环不变式
///
/// {用自然语言描述循环不变式，不要求 Lean 级别严格，但必须清晰可理解}
///
/// # 复杂度分析
///
/// - **时间复杂度**：{复杂度} — {简要说明}
/// - **空间复杂度**：{复杂度} — {简要说明}
///
/// # 证明要点（可选但推荐）
///
/// {关键正确性论证的简要说明}
pub fn algorithm_name(input: InputType) -> OutputType { ... }
```

### 3.2 边界测试用例（至少 3 个）

每个实现文件必须包含测试模块，至少覆盖：

1. **LeetCode 官方示例**（如有）
2. **空输入 / 最小规模输入**
3. **最大边界 / 极端值输入**
4. **特殊结构输入**（如全相同元素、已排序、逆序等）

示例（Rust）：

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_example_1() {
        assert_eq!(Solution::max_area(vec![1, 8, 6, 2, 5, 4, 8, 3, 7]), 49);
    }

    #[test]
    fn test_empty_or_minimal() {
        assert_eq!(Solution::max_area(vec![1, 1]), 1);
    }

    #[test]
    fn test_descending() {
        assert_eq!(Solution::max_area(vec![5, 4, 3, 2, 1]), 6);
    }
}
```

### 3.3 代码格式化

| 语言 | 格式化工具 | 检查命令 |
|------|-----------|---------|
| Rust | `rustfmt` | `cargo fmt --check` |
| Python | `black` / `ruff` | `black --check .` |
| Go | `gofmt` | `gofmt -d .` |
| TypeScript | `prettier` | `prettier --check .` |
| Java | `google-java-format` | — |
| C++ | `clang-format` | `clang-format --dry-run` |

## 4. 形式化规约注释规范

### 4.1 前置条件与后置条件

使用自然语言描述，不要求形式化逻辑表达式，但必须**无歧义**：

```rust
/// - **前置条件 (Pre)**：`nums` 已按升序排列，长度范围 `[1, 10^5]`，元素互不相同。
/// - **后置条件 (Post)**：若 `target ∈ nums`，返回唯一索引 `i` 使得 `nums[i] == target`；否则返回 `-1`。
```

### 4.2 循环不变式

在涉及循环的算法中，必须标注循环不变式：

```rust
/// # 循环不变式
///
/// 设当前搜索区间为 `[left, right]`：
/// - 若 `target` 在 `nums` 中，则 `target` 的索引必在 `[left, right]` 内。
/// - 每次迭代后，区间长度严格递减。
```

### 4.3 终止性说明（可选）

对于递归或复杂循环，建议补充终止性说明：

```rust
/// # 终止性
///
/// 递归深度不超过数组长度，且每次递归处理规模减半，故必终止。
```

## 5. 测试规范

### 5.1 测试文件组织

| 语言 | 测试文件位置 | 框架 |
|------|-------------|------|
| Rust | `src/` 内联 `#[cfg(test)] mod tests` 或 `tests/` 目录 | 内置 `cargo test` |
| Python | `tests/test_*.py` | `pytest` |
| Go | 与源码同目录 `*_test.go` | 内置 `go test` |
| TypeScript | `src/tests/*.ts` | `vitest` / `jest` |
| Java | `src/test/java/**/*.java` | JUnit |
| C++ | 同目录 `*_test.cpp` 或 `tests/` | Google Test / Catch2 |
| JavaScript | `tests/*.js` | `jest` / `mocha` |
| C | 同目录 `*_test.c` | `unity` / 自定义 |

### 5.2 覆盖率要求

- **每种语言的测试覆盖率应 ≥ 80%**
- 优先覆盖：
  1. 正常路径（Happy Path）
  2. 边界条件（空、单元素、最大值）
  3. 错误路径（无效输入、异常状态）

### 5.3 持续集成

建议在 CI 中配置以下检查：

```yaml
# .github/workflows/ci.yml 示例片段
- name: Run Rust tests
  run: cd examples/algorithms-rust && cargo test

- name: Run Python tests
  run: cd examples/algorithms-python && pytest --cov=src --cov-report=term-missing

- name: Run Go tests
  run: cd examples/algorithms-go && go test ./...
```

## 6. 跨语言一致性要求

### 6.1 核心思路一致

同一算法在不同语言中的实现，**核心思路、数据结构和复杂度必须保持一致**。允许的差异仅限于：

- 语言特有的语法糖（如 Rust 的 `match`，Python 的列表推导）
- 标准库差异（如 Java 的 `ArrayList` vs Rust 的 `Vec`）
- 内存管理方式（GC vs 手动 vs RAII）

### 6.2 复杂度标注一致

所有语言版本的同一算法，其 `Time Complexity` 和 `Space Complexity` 标注必须相同。

### 6.3 注释结构一致

所有语言版本的文档注释应包含相同的章节（前置条件、后置条件、循环不变式、复杂度分析），使用对应语言的文档注释语法：

| 语言 | 文档注释语法 |
|------|-------------|
| Rust | `///` |
| Python | `"""` |
| Go | `//` |
| TypeScript / JavaScript | `/** */` |
| Java | `/** */` |
| C++ | `/** */` |
| C | `/** */` |

## 7. 迁移与废弃策略

1. **新增实现**：优先在规范的语言目录中创建，禁止再在 `examples/algorithms/`（原始混杂目录）中添加新文件
2. **旧代码迁移**：参考本手册逐步将旧实现迁移到 Rosetta Code 模式
3. **废弃标记**：已迁移的旧目录保留 `README.md` 作为占位符，说明迁移去向
