# 季度内容审查清单 / Quarterly Content Review Checklist

> **文档版本**: 1.0
> **创建日期**: 2026-04-15
> **适用范围**: FormalAlgorithm 项目全文档库
> **执行频率**: 每季度第一个工作周

---

## 1. 引用时效性检查 / Citation Timeliness Check

### 1.1 检查目标

- 识别超过 **365 天**未更新的核心知识文档
- 重点关注含有 `last_updated` 元数据或版本声明的 `.md` 文件

### 1.2 检查范围

| 模块路径 | 权重 | 阈值 |
|---------|------|------|
| `docs/01-基础理论/` | 高 | >365 天 |
| `docs/02-递归理论/` | 高 | >365 天 |
| `docs/04-算法复杂度/` | 高 | >365 天 |
| `docs/05-类型理论/` | 高 | >365 天 |
| `docs/07-计算模型/` | 高 | >365 天 |
| `docs/09-算法理论/` | 高 | >365 天 |
| `docs/10-高级主题/` | 中 | >365 天 |
| `docs/12-应用领域/` | 中 | >365 天 |

### 1.3 自动化工具

- 运行 `scripts/quarterly_review.py` 生成过期文档列表
- 输出位置: `reports/quarterly_review_YYYY-MM-DD.md`

### 1.4 人工复核项

- [ ] 检查引用数据库 (`docs/references_database_2024_2025.yaml`) 中是否有新增文献未被任何文档引用
- [ ] 核对 arXiv/会议论文的发表日期，更新过时的理论引用
- [ ] 确认所有外部链接（Wikipedia、课程主页、开源仓库）仍可访问

## 3. 代码示例编译检查 / Code Example Compilation Check

### 3.1 Rust 示例

- [ ] 进入 `examples/algorithms/`，运行 `cargo test`
- [ ] 检查是否有因文档与代码不同步导致的编译失败

### 3.2 Lean 示例

- [ ] 进入 `examples/algorithms/`（或专门的 Lean 项目目录），运行 `lake build` 或 `lean --make`
- [ ] 确认形式化证明片段在最新 Lean 版本中可通过

### 3.3 其他语言

- [ ] 对 `examples/algorithms-c/` 运行 `make`（如有 Makefile）
- [ ] 对 `examples/algorithms-cpp/` 运行 `cmake --build`
- [ ] 对 `examples/algorithms-python/` 运行 `pytest`

### 3.4 记录与修复

- 将编译错误截图或日志附到季度审查报告中
- 指派对应语言工作组的 owner 在 2 周内修复

## 4. Wiki 对齐验证检查 / Wiki Alignment Verification

### 4.1 检查目标

- 确保项目关键概念与 Wikipedia/标准术语的定义保持一致
- 验证 `docs/11-国际化/` 下的 Wiki 对齐表无过期链接

### 4.2 检查清单

- [ ] 随机抽样 10 个核心概念，核对其 Wiki 链接是否指向最新版本
- [ ] 检查 `docs/11-国际化/02-Wiki国际概念对齐.md` 及补充文档中的外部链接有效性
- [ ] 确认新增概念已在 `docs/术语与符号总表-v2.md` 和 `docs/知识图谱/concept_nodes.yaml` 中注册

### 4.3 术语一致性

- [ ] 检查是否存在同一英文术语对应多个中文译名的情况
- [ ] 核对数学符号的使用是否与 `docs/数学符号标准化规范.md` 一致

## 5. 审查报告模板 / Review Report Template

每季度审查完成后，需在 `reports/` 目录下生成报告，命名格式：

```
reports/quarterly_review_YYYY-MM-DD.md
```

报告至少包含：

1. 执行摘要（发现问题数、已修复数、遗留问题）
2. 过期文档列表（按模块分组）
3. 未引用文献列表（来自引用数据库）
4. 编译检查结果（PASS / FAIL 汇总）
5. Wiki 对齐问题汇总
6. 下季度行动计划
