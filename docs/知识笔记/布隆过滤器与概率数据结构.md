## 一、核心概念

### 1.1 定义

**正式定义**:
```
布隆过滤器（Bloom Filter）是由 Burton Howard Bloom 于 1970 年提出的概率型数据结构，
用于高效地测试一个元素是否属于一个集合。它是一个长度为 m 的位数组 B[0..m-1]，
配合 k 个独立的哈希函数 h₁, h₂, ..., hₖ: U → {0, 1, ..., m-1}，其中 U 是全集。

形式化描述为一个四元组 BF = (B, k, H, n)：
- B: 位数组，|B| = m
- k: 哈希函数数量
- H = {h₁, ..., hₖ}: 哈希函数族
- n: 预期插入元素数量

核心性质：
- 假阴性率（False Negative Rate）: 0%（若元素在集合中，必返回"可能存在"）
- 假阳性率（False Positive Rate）: ε > 0（若元素不在集合中，可能误判为"可能存在"）
```

**直观解释**:
布隆过滤器就像一张"可以通过指纹快速判断某人是否可能到过现场"的档案。如果某人的指纹在档案中，他一定来过；但如果不在，他仍有可能来过却未留指纹。这是一种以"可能漏报"来换取"绝不误杀"和"极度省空间"的智慧数据结构。非常适合用于"快速排除"场景——如缓存穿透防护、数据库查询优化等。

**关键要点**:
- **空间效率极高**: 每个元素仅需约 1.44·log₂(1/ε) 位，与元素大小无关
- **只增不减**: 标准布隆过滤器不支持删除操作（Counting Bloom Filter 除外）
- **哈希函数独立**: k 个哈希函数需相互独立且均匀分布
- **无法获取元素**: 布隆过滤器只回答"是否存在"，不存储元素本身

### 1.2 属性

| 属性 | 描述 | 备注 |
|------|------|------|
| 空间复杂度 | O(m) = O(n·log₂(1/ε)) | m 为位数组长度，n 为元素数，ε 为假阳性率 |
| 插入时间 | O(k) | 需计算 k 个哈希值并设置对应位 |
| 查询时间 | O(k) | 需计算 k 个哈希值并检查对应位 |
| 假阴性率 | 0 | 绝对不存在假阴性 |
| 假阳性率 | ≈ (1 - e^(-kn/m))^k | 理论计算公式 |
| 可删除性 | 否（标准版） | Counting Bloom Filter 支持删除 |

**性质总结**:
1. **概率完备性**: 若查询返回"不存在"，则元素绝对不在集合中；若返回"可能存在"，则有一定概率是误判
2. **单调性**: 随着插入元素增多，假阳性率单调递增
3. **不可恢复性**: 一旦位被置 1，无法确定是哪个元素设置的，因此无法逆向获取元素

### 1.3 变体

**Counting Bloom Filter (计数布隆过滤器)**:
- 定义: 将位数组的每个位扩展为计数器（通常 4 位），记录该位置被哈希到的次数
- 与标准形式的区别: 支持删除操作（对应计数器减 1），但空间开销增加 4 倍
- 适用场景: 需要动态删除元素的场景，如网络缓存、动态集合管理

**Cuckoo Filter (布谷鸟过滤器)**:
- 定义: 基于布谷鸟哈希（Cuckoo Hashing）的过滤器，存储元素的指纹（fingerprint）
- 与标准形式的区别: 支持删除、查找效率更高（只需检查 2 个位置）、假阳性率可配置
- 适用场景: 对查询性能要求极高且需要删除功能的场景

**Quotient Filter (商过滤器)**:
- 定义: 基于罗宾汉哈希（Robin Hood Hashing）的过滤器，使用商-余数编码存储指纹
- 与标准形式的区别: 支持合并操作、更好的缓存局部性、可动态扩容
- 适用场景: 需要高效合并多个过滤器的分布式系统

**Blocked Bloom Filter (分块布隆过滤器)**:
- 定义: 将位数组划分为多个块，每个元素映射到一个块内的位
- 与标准形式的区别: 更好的缓存性能，但理论假阳性率略高
- 适用场景: 高性能缓存系统

## 三、形式化证明

### 3.1 核心定理

**定理 1 (假阳性率上界)**: 对于包含 n 个元素的布隆过滤器，使用 m 位和 k 个哈希函数，假阳性率为：

$$
\varepsilon \approx \left(1 - e^{-kn/m}\right)^k
$$

**证明**:

**步骤 1: 计算某一位仍为 0 的概率**

对于特定的一位，单个哈希函数不将其置 1 的概率为 $1 - \frac{1}{m}$。

由于有 k 个哈希函数，插入一个元素时该位不被置 1 的概率为：

$$
P(\text{位仍为0} | \text{插入1个元素}) = \left(1 - \frac{1}{m}\right)^k
$$

**步骤 2: 插入 n 个元素后的概率**

假设哈希函数均匀且独立，插入 n 个元素后该位仍为 0 的概率：

$$
P(\text{位为0}) = \left(1 - \frac{1}{m}\right)^{kn}
$$

当 m 很大时，使用近似 $(1 - 1/m)^m \approx e^{-1}$：

$$
P(\text{位为0}) = \left[\left(1 - \frac{1}{m}\right)^m\right]^{kn/m} \approx e^{-kn/m}
$$

**步骤 3: 某一位为 1 的概率**

$$
P(\text{位为1}) = 1 - e^{-kn/m}
$$

**步骤 4: 假阳性事件**

对于不在集合中的元素 x，假阳性发生当且仅当 x 的 k 个哈希位置都已经被置 1：

$$
\varepsilon = P(\text{假阳性}) = \left(1 - e^{-kn/m}\right)^k
$$

**证明要点分析**:
1. **独立性假设**: 证明依赖于哈希函数的均匀性和独立性假设
2. **泊松近似**: 使用 $e^{-x}$ 近似 $(1-x/n)^n$，在 n,m 很大时准确
3. **联合概率**: 假阳性需要 k 个独立事件同时发生

**直觉理解**:
假阳性率取决于"位被填满的程度"。当位数组越满（kn/m 越大），随机查询撞上一串已置位的概率就越高。增加哈希函数 k 在初期降低假阳性率（更多位被检查），但超过某点后反而增加假阳性率（位数组被填满得更快）。

---

**定理 2 (最优哈希函数数量)**: 给定 m 和 n，使假阳性率最小的最优哈希函数数量为：

$$
k_{opt} = \frac{m}{n} \ln 2 \approx 0.693 \cdot \frac{m}{n}
$$

**证明**:

设 $p = e^{-kn/m}$，则假阳性率 $\varepsilon = (1-p)^k$。

令 $g = k \cdot \ln(1-p) = k \cdot \ln(1 - e^{-kn/m})$。

为求最小值，对 k 求导并令导数为 0：

$$
\frac{dg}{dk} = \ln(1-p) + k \cdot \frac{1}{1-p} \cdot \frac{dp}{dk} = 0
$$

其中 $\frac{dp}{dk} = -\frac{n}{m} \cdot e^{-kn/m} = -\frac{n}{m} \cdot p$。

代入并求解可得 $k_{opt} = \frac{m \ln 2}{n}$。

---

**定理 3 (最优空间与假阳性率关系)**: 给定假阳性率 $\varepsilon$，最小空间需求为：

$$
m = -\frac{n \ln \varepsilon}{(\ln 2)^2} \approx 1.44 \cdot n \cdot \log_2\left(\frac{1}{\varepsilon}\right)
$$

**证明**:

将 $k_{opt} = \frac{m}{n}\ln 2$ 代入假阳性率公式：

$$
\varepsilon = \left(1 - e^{-\ln 2}\right)^{\frac{m}{n}\ln 2} = \left(\frac{1}{2}\right)^{\frac{m}{n}\ln 2}
$$

取对数并解出 m 即得结果。

---

### 3.2 辅助引理

**引理 1 (哈希独立性假设)**: 若 k 个哈希函数是 2-独立的（pairwise independent），则假阳性率的上界为 $(1/m + \varepsilon_{hash})^k$，其中 $\varepsilon_{hash}$ 是哈希函数的非理想性偏差。

**引理 2 (布隆过滤器的并集界)**: 两个布隆过滤器 $BF_1$ 和 $BF_2$ 的按位或操作产生的新过滤器等价于两个集合的并集的布隆过滤器。

## 四、算法/方法详解

### 4.1 算法描述

**伪代码**:
```
数据结构: BloomFilter(m, k)
    B: 位数组[0..m-1], 初始全为 0
    H: 哈希函数族 {h₁, h₂, ..., hₖ}

算法: Insert(x)
输入: 元素 x
输出: 无（副作用：修改过滤器）

1. for i = 1 to k do
2.     j ← hᵢ(x) mod m
3.     B[j] ← 1
4. end for

算法: Query(x)
输入: 元素 x
输出: "可能存在" 或 "肯定不存在"

1. for i = 1 to k do
2.     j ← hᵢ(x) mod m
3.     if B[j] = 0 then
4.         return "肯定不存在"
5.     end if
6. end for
7. return "可能存在"

算法: OptimalParameters(n, ε)
输入: 预期元素数 n, 目标假阳性率 ε
输出: 最优参数 (m, k)

1. m ← ⌈-n · ln(ε) / (ln 2)²⌉
2. k ← ⌈(m/n) · ln 2⌉
3. return (m, k)
```

### 4.2 正确性分析

**不变式**: 对于布隆过滤器中的每一位 B[i]，若 B[i] = 1，则存在至少一个已插入的元素 x 和某个哈希函数 $h_j$ 使得 $h_j(x) \equiv i \pmod{m}$。

**证明**:
- **初始化**: 初始时所有位为 0，不变式自然成立
- **保持**: Insert(x) 操作只将位设置为 1，设置 B[j]=1 时对应 $h_i(x) = j$，不变式保持
- **终止**: 查询操作不修改位数组，终止时不变式仍然成立

### 4.3 复杂度分析

**时间复杂度**:
- 最坏情况: $O(k)$ - 需计算 k 个哈希值
- 最好情况: $O(1)$ - 查询时第一个哈希位置为 0 可立即返回
- 平均情况: $O(k)$ - 假设均匀分布

**空间复杂度**: $O(m) = O(n \log(1/\varepsilon))$

**空间效率对比**:
- 精确集合（哈希表）: $O(n \cdot |x|)$ 位
- 布隆过滤器: $O(n \log(1/\varepsilon))$ 位，与 $|x|$ 无关

当 $|x| \gg \log(1/\varepsilon)$ 时，布隆过滤器空间优势明显。

## 五、示例与实例

### 5.1 标准示例

**示例 1**: 构建一个布隆过滤器存储 1000 个 URL，要求假阳性率低于 1%

**解决过程**:
1. 参数计算：
   - n = 1000, ε = 0.01
   - m = ⌈-1000 × ln(0.01) / (ln 2)²⌉ = ⌈9585.06⌉ = 9586 位 ≈ 1.2 KB
   - k = ⌈(9586/1000) × ln 2⌉ = ⌈6.64⌉ = 7 个哈希函数

2. 预期性能：
   - 假阳性率 ≈ (1 - e^(-7×1000/9586))^7 ≈ 0.0082 < 0.01
   - 对比：哈希表存储 1000 个平均 50 字节的 URL 需约 50 KB

**结果**: 空间节省约 40 倍，查询时间 O(7) = O(1)

### 5.2 代码实现

**Python 实现**:

```python
import mmh3  # MurmurHash3
import math
from bitarray import bitarray

class BloomFilter:
    def __init__(self, expected_items: int, false_positive_rate: float):
        """
        初始化布隆过滤器
        
        Args:
            expected_items: 预期插入元素数量
            false_positive_rate: 目标假阳性率 (0, 1)
        """
        self.n = expected_items
        self.epsilon = false_positive_rate
        
        # 计算最优参数
        self.m = math.ceil(-self.n * math.log(self.epsilon) / (math.log(2) ** 2))
        self.k = math.ceil(self.m / self.n * math.log(2))
        
        # 初始化位数组
        self.bit_array = bitarray(self.m)
        self.bit_array.setall(0)
        self.item_count = 0
    
    def _get_hash_positions(self, item: str):
        """计算元素的所有哈希位置（使用双重哈希技术）"""
        positions = []
        # 使用两个独立的哈希函数生成 k 个位置
        hash1 = mmh3.hash(item, 0) % self.m
        hash2 = mmh3.hash(item, 1) % self.m
        
        for i in range(self.k):
            # 双重哈希: (h1 + i * h2) % m
            pos = (hash1 + i * hash2) % self.m
            positions.append(pos)
        return positions
    
    def add(self, item: str):
        """插入元素"""
        for pos in self._get_hash_positions(item):
            self.bit_array[pos] = 1
        self.item_count += 1
    
    def __contains__(self, item: str) -> bool:
        """查询元素是否可能存在"""
        for pos in self._get_hash_positions(item):
            if self.bit_array[pos] == 0:
                return False
        return True
    
    def current_false_positive_rate(self) -> float:
        """计算当前假阳性率"""
        return (1 - math.exp(-self.k * self.item_count / self.m)) ** self.k
    
    def __repr__(self):
        return f"BloomFilter(m={self.m}, k={self.k}, n={self.item_count}, ε={self.current_false_positive_rate():.6f})"


# 使用示例
if __name__ == "__main__":
    # 创建布隆过滤器：预期 10000 个元素，1% 假阳性率
    bf = BloomFilter(expected_items=10000, false_positive_rate=0.01)
    print(f"初始化: {bf}")
    
    # 添加元素
    urls = [
        "https://example.com/page1",
        "https://example.com/page2",
        "https://example.com/page3"
    ]
    for url in urls:
        bf.add(url)
    
    # 查询
    print(f"'page1' in bf: {'https://example.com/page1' in bf}")  # True
    print(f"'page4' in bf: {'https://example.com/page4' in bf}")  # False (大概率)
    print(f"当前状态: {bf}")
```

**Rust 实现**:

```rust
use std::collections::hash_map::DefaultHasher;
use std::hash::{Hash, Hasher};
use bitvec::vec::BitVec;

pub struct BloomFilter {
    bit_array: BitVec,
    m: usize,        // 位数组大小
    k: usize,        // 哈希函数数量
    n: usize,        // 已插入元素数
}

impl BloomFilter {
    /// 创建新的布隆过滤器
    pub fn new(expected_items: usize, false_positive_rate: f64) -> Self {
        let m = Self::optimal_m(expected_items, false_positive_rate);
        let k = Self::optimal_k(expected_items, m);
        
        BloomFilter {
            bit_array: BitVec::repeat(false, m),
            m,
            k,
            n: 0,
        }
    }
    
    /// 计算最优位数组大小
    fn optimal_m(n: usize, epsilon: f64) -> usize {
        let m = -(n as f64) * epsilon.ln() / (2.0f64.ln().powi(2));
        m.ceil() as usize
    }
    
    /// 计算最优哈希函数数量
    fn optimal_k(n: usize, m: usize) -> usize {
        let k = (m as f64 / n as f64) * 2.0f64.ln();
        k.ceil() as usize
    }
    
    /// 双重哈希：使用两个种子生成 k 个哈希值
    fn hash_positions<T: Hash>(&self, item: &T) -> Vec<usize> {
        let mut positions = Vec::with_capacity(self.k);
        
        let mut hasher1 = DefaultHasher::new();
        item.hash(&mut hasher1);
        let hash1 = hasher1.finish() as usize;
        
        let mut hasher2 = DefaultHasher::new();
        (item, 0x12345678u32).hash(&mut hasher2);
        let hash2 = hasher2.finish() as usize;
        
        for i in 0..self.k {
            let pos = (hash1.wrapping_add(i.wrapping_mul(hash2))) % self.m;
            positions.push(pos);
        }
        positions
    }
    
    /// 插入元素
    pub fn insert<T: Hash>(&mut self, item: &T) {
        for pos in self.hash_positions(item) {
            self.bit_array.set(pos, true);
        }
        self.n += 1;
    }
    
    /// 查询元素
    pub fn contains<T: Hash>(&self, item: &T) -> bool {
        self.hash_positions(item)
            .iter()
            .all(|&pos| self.bit_array[pos])
    }
}
```

**代码说明**:
- **双重哈希**: 使用 2 个独立哈希函数生成 k 个位置，避免存储 k 个哈希函数
- **bitarray/bitvec**: 高效的位数组实现，支持直接位操作
- **类型泛型**: Rust 实现支持任意可哈希类型

## 5.3 反例
### 5.3 反例

**常见错误**:

**错误 1**: 删除操作导致假阴性
```python
# 错误代码：尝试删除元素
class BrokenBloomFilter(BloomFilter):
    def remove(self, item: str):
        # 错误：将对应位清零
        for pos in self._get_hash_positions(item):
            self.bit_array[pos] = 0  # ❌ 危险操作！

# 场景：
bf = BrokenBloomFilter(100, 0.01)
bf.add("A")  # 设置位 [3, 7, 12]
bf.add("B")  # 设置位 [7, 15, 20]（位 7 与 A 冲突）
bf.remove("A")  # 清零位 [3, 7, 12]

# 现在查询 B：
print("B" in bf)  # False！假阴性！因为位 7 被清零
```
**错误原因**: 多个元素可能共享同一个位，删除一个元素会影响其他元素
**正确做法**: 使用 Counting Bloom Filter，维护计数器而非简单位

**错误 2**: 超过设计容量导致假阳性率飙升
```python
# 设计容量：100 个元素，1% 假阳性率
bf = BloomFilter(100, 0.01)

# 错误：插入 1000 个元素（10倍容量）
for i in range(1000):
    bf.add(f"item_{i}")

# 实际假阳性率
print(f"设计假阳性率: 0.01")
print(f"实际假阳性率: {bf.current_false_positive_rate():.4f}")  # 约 0.58！
```
**错误原因**: 位数组过满，几乎所有位都被置 1
**正确做法**: 监控元素数量，必要时重建过滤器或使用 Scalable Bloom Filter

## 六、习题

### 6.1 理解题

1. **布隆过滤器的假阴性率是多少？为什么？** [难度⭐]
   
   <details>
   <summary>解答</summary>
   
   **假阴性率为 0（不存在假阴性）**。
   
   原因：布隆过滤器的查询逻辑是"检查所有 k 个哈希位置，若任一位为 0 则返回不存在"。若元素确实被插入过，其 k 个位置在插入时必定被置 1，且布隆过滤器不支持删除（标准版），因此这些位永远为 1。查询时必返回"可能存在"，不会出现"不存在"的错误判断。
   
   </details>

2. **为什么布隆过滤器的空间效率与元素大小无关？** [难度⭐⭐]
   
   <details>
   <summary>解答</summary>
   
   布隆过滤器只存储哈希值的位标记，不存储元素本身。无论元素是 10 字节的字符串还是 1MB 的文件，经过哈希函数后都映射为固定范围的整数（0 到 m-1）。因此空间复杂度仅取决于期望的假阳性率 ε 和预期元素数量 n，与元素实际大小无关。这使得布隆过滤器特别适合存储大对象的存在性信息。
   
   </details>

### 6.2 应用题

1. **设计一个 URL 去重系统** [难度⭐⭐]
   
   需要处理 1 亿个 URL，平均长度 100 字节，内存限制 100MB，允许最多 0.1% 的误判率。请设计布隆过滤器参数并计算内存节省。
   
   <details>
   <summary>解答</summary>
   
   **参数设计**:
   - n = 100,000,000, ε = 0.001
   
   **计算**:
   ```
   m = -n·ln(ε) / (ln 2)²
     = -10⁸ · ln(0.001) / 0.4804
     = 10⁸ · 6.907 / 0.4804
     ≈ 1.44 × 10⁹ 位 ≈ 180 MB
   ```
   
   这超出了 100MB 限制。需要调整参数：
   
   **优化方案**: 使用分片布隆过滤器：每 5000 万个 URL 用一个过滤器，或使用两个层级的过滤器结构。
   
   **内存节省对比**:
   - 精确存储（哈希表）：100M × 100B = 10 GB
   - 布隆过滤器（ε=0.1%）：180 MB
   - 节省比例：约 55 倍
   
   </details>

2. **布隆过滤器合并** [难度⭐⭐⭐]
   
   分布式系统中有三个节点，各自维护一个布隆过滤器。请设计算法合并这些过滤器，并证明合并后的假阳性率上界。
   
   <details>
   <summary>解答</summary>
   
   **合并算法**:
   ```python
   def merge_bloom_filters(bf1, bf2):
       assert bf1.m == bf2.m and bf1.k == bf2.k
       merged = BloomFilter(bf1.m, bf1.k)
       for i in range(bf1.m):
           merged.bit_array[i] = bf1.bit_array[i] | bf2.bit_array[i]
       return merged
   ```
   
   **正确性证明**:
   设 $S_1$, $S_2$ 为两个集合，$BF_1$, $BF_2$ 为对应过滤器。
   
   对于任意元素 x：$x \in S_1 \cup S_2$ ⟺ $x \in S_1$ 或 $x \in S_2$ ⟺ 按位或后的过滤器中 x 对应位被置位。
   
   **假阳性率上界**: 合并后的假阳性率不超过各过滤器假阳性率之和：$\varepsilon_{merged} \leq \varepsilon_1 + \varepsilon_2$
   
   </details>

### 6.3 证明题

1. **证明最优 k 值下假阳性率为 (1/2)^k** [难度⭐⭐⭐]
   
   证明当使用最优哈希函数数量 $k = (m/n)\ln 2$ 时，假阳性率恰好为 $(1/2)^k$。
   
   <details>
   <summary>解答</summary>
   
   **证明**:
   
   假阳性率公式：$\varepsilon = (1 - e^{-kn/m})^k$
   
   代入 $k = (m/n)\ln 2$：
   
   $$e^{-kn/m} = e^{-\ln 2} = 1/2$$
   
   因此：$\varepsilon = (1 - 1/2)^k = (1/2)^k$
   
   证毕。
   
   </details>

| 推荐系统 | 已推荐内容过滤 | 快速判断内容是否已推荐 |

### 7.2 实际案例

**案例**: Google Chrome 恶意 URL 检测

**背景**:
Google Chrome 需要保护用户免受恶意网站侵害。恶意 URL 数据库庞大（数十亿条），但客户端无法存储完整列表。

**应用方式**:
1. 服务器维护完整恶意 URL 列表
2. 定期生成布隆过滤器（约 30MB），包含所有恶意 URL 的指纹
3. 客户端下载布隆过滤器
4. 用户访问网站时，先本地查询布隆过滤器
   - 若返回"不存在": 安全访问
   - 若返回"可能存在": 向服务器确认（真阳性）或放行（假阳性）

**效果**:
- 99.99% 的查询在本地完成（假阳性率设计为 0.01%）
- 客户端仅需 30MB 存储，对比完整列表需数 GB
- 查询延迟从网络往返（~100ms）降至本地计算（~1μs）

---

**案例**: LevelDB/RocksDB 的 SSTable 过滤

**背景**:
LSM-Tree 结构的键值存储（如 LevelDB、RocksDB）将数据分层存储，每层包含多个 SSTable 文件。查询时需要确定 key 可能存在于哪个 SSTable。

**应用方式**:
1. 每个 SSTable 文件附带一个布隆过滤器
2. 查询时先检查布隆过滤器，确定需要读取的 SSTable
3. 仅需读取可能包含 key 的 SSTable，避免全量扫描

**效果**:
- 对于不存在的 key 查询，磁盘 I/O 从 O(文件数) 降至 O(1)
- 布隆过滤器常驻内存，查询开销可忽略

### 7.3 跨领域联系

**与分布式系统的联系**:
分布式数据库（如 Cassandra、HBase）使用布隆过滤器优化跨节点的数据定位。每个节点维护本地布隆过滤器，协调节点通过过滤器快速确定查询应路由到哪些节点。

**与网络安全的联系**:
- **入侵检测系统 (IDS)**: 使用布隆过滤器存储已知恶意签名，快速匹配网络流量
- **密码泄露检测**: Have I Been Pwned 等服务使用布隆过滤器让客户端本地检查密码是否可能泄露

## 八、多维对比

### 8.1 方法对比

| 维度 | 布隆过滤器 | Cuckoo 过滤器 | Quotient 过滤器 | 哈希表 |
|------|------------|---------------|-----------------|--------|
| 空间复杂度 | O(n log(1/ε)) | O(n log(1/ε)) | O(n log(1/ε)) | O(n) |
| 查询时间 | O(k) | O(1) | O(1) | O(1) 平均 |
| 支持删除 | 否（需计数版） | 是 | 是 | 是 |
| 可扩展性 | 差（需重建） | 差 | 好 | 好 |
| 缓存友好性 | 一般 | 好 | 好 | 好 |
| 假阳性率 | 可调 | 可调 | 可调 | 0 |

### 8.2 概率数据结构对比

| 数据结构 | 功能 | 空间复杂度 | 误差类型 | 典型应用 |
|----------|------|------------|----------|----------|
| **Bloom Filter** | 成员查询 | O(n log(1/ε)) | 假阳性 | 缓存穿透防护 |
| **HyperLogLog** | 基数估计 | O(log log n) | 相对误差 ~2% | UV 统计 |
| **Count-Min Sketch** | 频率估计 | O((1/ε) log(1/δ)) | 高估 | 流量监控 |
| **MinHash** | 相似度估计 | O(k) | Jaccard 估计 | 文档去重 |
| **Top-K** | 频繁项 | O(k) | 可能遗漏 | 热门内容 |

**HyperLogLog 详解**:
用于基数估计（集合中不同元素的数量）。空间复杂度仅 O(log log n)，可处理数十亿级别的基数估计，误差约 1-2%。应用于网站 UV 统计、数据库 DISTINCT 优化等。

**Count-Min Sketch 详解**:
用于频率估计（元素出现次数）。使用 d × w 的计数器矩阵，通过 d 个哈希函数映射。保证估计值不小于真实值（只高估不低估）。应用于网络流量监控、热门内容统计等。

**MinHash 详解**:
用于估计两个集合的 Jaccard 相似度。通过 k 个哈希函数生成签名，比较签名估计相似度。应用于文档去重、近似最近邻搜索等。

**决策建议**:

**何时选择布隆过滤器**:
- 只需要判断"是否存在"
- 空间极度受限
- 不需要删除操作或删除频率低

**何时选择 Cuckoo 过滤器**:
- 需要支持删除
- 查询性能要求极高
- 可以接受稍高的实现复杂度

**决策流程图**:
```
需要概率数据结构?
    │
    ├── 需要基数估计? ──▶ HyperLogLog
    │
    ├── 需要频率估计? ──▶ Count-Min Sketch
    │
    ├── 需要相似度估计? ──▶ MinHash
    │
    └── 需要成员查询?
            │
            ├── 需要删除? ──▶ Cuckoo Filter / Quotient Filter
            │
            └── 不需要删除? ──▶ Bloom Filter
                    │
                    ├── 需要合并多个过滤器? ──▶ Quotient Filter
                    │
                    └── 标准场景? ──▶ 标准 Bloom Filter
```
