## 一、核心概念

### 1.1 定义

**正式定义**:
流算法处理的是**数据流**（data stream）——一个长度可能无限的输入序列 $a_1, a_2, ..., a_m$，其中每个元素来自 Universe $[n] = \{1, 2, ..., n\}$。算法只能：
- 按顺序扫描数据流一次（或有限次）
- 使用远小于输入规模的内存空间（通常为 $O(\log n)$ 或 $O(\text{polylog}(n))$）
- 在有限时间内处理每个元素

**流模型变体**:
| 模型 | 描述 | 适用场景 |
|------|------|----------|
| **Cash Register Model** | 仅插入（$x_i \geq 0$） | 网络流量统计 |
| **Turnstile Model** | 插入和删除（$x_i \in \mathbb{Z}$） | 数据库事务 |
| **Strict Turnstile** | 插入和删除，保证 $x_i \geq 0$ | 滑动窗口统计 |
| **Sliding Window** | 只关注最近 $W$ 个元素 | 实时监控 |

**直观解释**:
想象你在统计一条繁忙高速公路上的车辆类型，但只能使用一个小笔记本记录。你无法记住所有经过的车辆，但必须能回答"今天经过了多少辆红色轿车？"这类问题。流算法就是设计精巧的"笔记本记录方法"。

**关键要点**:
- **亚线性空间**: 内存使用 $o(\min(n, m))$，通常 $O(\text{polylog}(nm))$
- **单遍扫描**: 数据只读一次，无法回头
- **近似保证**: 由于空间限制，通常提供 $(\epsilon, \delta)$-近似
- **实时处理**: 每个元素处理时间必须很低

### 1.2 属性

| 属性 | 描述 | 备注 |
|------|------|------|
| 近似比 | $(1\pm\epsilon)$-近似 | $\epsilon$ 为相对误差 |
| 成功概率 | $1-\delta$ | 通过重复降低失败概率 |
| 空间复杂度 | 通常 $O(\epsilon^{-2} \log \delta^{-1} \log n)$ | 依赖误差和宇宙大小 |
| 更新时间 | $O(1)$ 或 $O(\log n)$ | 摊还或最坏情况 |

**性质总结**:
1. **空间-精度权衡**: 更高精度需要更多空间，通常与 $1/\epsilon^2$ 成正比
2. **不可计算性**: 某些问题（如精确中位数）需要 $\Omega(n)$ 空间
3. **随机化必要性**: 许多问题确定性算法需要线性空间

### 1.3 变体

**单遍 vs 多遍**:
- 定义: 允许扫描数据流的次数
- 与标准形式的区别: 多遍算法空间可能指数级减少
- 适用场景: 外部存储、大规模数据库处理

**分布式流**:
- 定义: 多节点分别处理流片段，定期通信
- 与标准形式的区别: 考虑通信复杂度
- 适用场景: 传感器网络、分布式监控

**图流**:
- 定义: 流中的元素是图的边
- 与标准形式的区别: 关注图结构性质
- 适用场景: 社交网络、网络拓扑分析

## 三、形式化证明

### 3.1 核心定理

**定理 1** (Count-Min Sketch的近似保证):
设 $d = \lceil \ln(1/\delta) \rceil$, $w = \lceil e/\epsilon \rceil$。Count-Min Sketch 提供频率估计 $\hat{f}_i$ 满足：
$$f_i \leq \hat{f}_i \leq f_i + \epsilon \|f\|_1$$
概率至少 $1-\delta$，其中 $\|f\|_1 = \sum_j f_j$ 是总频率。

**证明**:

设哈希函数 $h_1, ..., h_d: [n] \to [w]$ 从2-通用哈希族中独立选取。

对于计数器 $C[j][h_j(i)]$，其值为：
$$C[j][h_j(i)] = f_i + \sum_{k \neq i, h_j(k) = h_j(i)} f_k$$

令 $X_{j,k} = \mathbf{1}_{h_j(k) = h_j(i)}$ 为指示变量。由2-通用性：
$$E[X_{j,k}] = \frac{1}{w}$$

估计误差：
$$\hat{f}_i - f_i = \min_j \sum_{k \neq i} f_k \cdot X_{j,k}$$

对固定 $j$：
$$E\left[\sum_{k \neq i} f_k \cdot X_{j,k}\right] = \frac{\|f\|_1 - f_i}{w} \leq \frac{\|f\|_1}{w} \leq \frac{\epsilon \|f\|_1}{e}$$

由马尔可夫不等式：
$$P\left(\sum_{k \neq i} f_k \cdot X_{j,k} \geq \epsilon \|f\|_1\right) \leq \frac{1}{e}$$

所有 $d$ 个哈希函数都失败的概率：
$$P(\hat{f}_i - f_i \geq \epsilon \|f\|_1) \leq \left(\frac{1}{e}\right)^d \leq \delta$$

因此：$P(f_i \leq \hat{f}_i \leq f_i + \epsilon \|f\|_1) \geq 1-\delta$。∎

**证明要点分析**:
1. **2-通用哈希**: 保证哈希碰撞的期望可控
2. **最小估计**: 取 $d$ 个估计的最小值减少方差
3. **空间-精度权衡**: 空间 $O((1/\epsilon)\log(1/\delta))$

### 3.2 辅助引理

**引理 1** (Distinct Elements的空间下界):
精确计算 $m$ 个元素中不同元素的数量需要 $\Omega(n)$ 空间（确定性算法）或 $\Omega(\log n)$ 空间（随机化算法达到常数近似）。

*证明概要*:
通过通信复杂度归约。假设 Alice 和 Bob 分别持有集合 $A$ 和 $B$，他们想判断 $A \cap B$ 是否为空。设 $m = 2n$，将问题编码为流：先输入 $A$ 的元素，再输入 $B$ 的元素。不同元素数量等于 $|A \cup B| = |A| + |B| - |A \cap B|$。若能在小空间近似不同元素数量，则可解决集合不交问题，这需要 $\Omega(n)$ 通信。∎

## 四、算法/方法详解

### 4.1 Count-Min Sketch算法

**伪代码**:
```
算法: Count-Min Sketch
输入: 数据流，宇宙大小 n，误差参数 ε，失败概率 δ

1.  w ← ⌈e/ε⌉, d ← ⌈ln(1/δ)⌉
2.  初始化 C[d][w] ← 0
3.  从2-通用哈希族选取 h₁, ..., h_d: [n] → [w]
4.  
5.  对每个元素 x 在流中:
6.      for j = 1 to d do
7.          C[j][h_j(x)] ← C[j][h_j(x)] + 1
8.      end for
9.  
10. 查询频率(x):
11.     return min_{j=1..d} C[j][h_j(x)]
```

**流程图**:
```
流元素 x → 哈希函数 h₁,...,h_d → 更新 C[1][h₁(x)],...,C[d][h_d(x)]
                ↓
查询 y → 计算 min_j C[j][h_j(y)] → 输出估计频率
```

### 4.2 正确性分析

**不变式**: 对所有 $j, k$，$C[j][k] = \sum_{x: h_j(x)=k} f_x$

**证明**:
- **初始化**: 所有计数器为0，成立
- **保持**: 处理元素 $x$ 时，对所有 $j$，$C[j][h_j(x)]$ 增加1，对应 $f_x$ 增加1
- **终止**: 流处理完毕后，不变式仍成立

### 4.3 复杂度分析

**空间复杂度**: $O((1/\epsilon) \cdot \log(1/\delta) \cdot \log m)$ 位
- $w \times d$ 个计数器
- 每个计数器需 $O(\log m)$ 位（最大频率）

**时间复杂度**:
- 更新: $O(\log(1/\delta))$（计算 $d$ 个哈希值）
- 查询: $O(\log(1/\delta))$

**近似保证**:
- 单边误差：$f_i \leq \hat{f}_i \leq f_i + \epsilon \|f\|_1$
- 成功概率：$1-\delta$

## 五、示例与实例

### 5.1 标准示例

**示例 1**: 网络流量统计

**问题描述**:
监控网络中数据包的目的IP地址，估计每个IP的流量。
- Universe: IPv4地址空间 $n = 2^{32}$
- 每日数据包数: $m \approx 10^{12}$
- 要求: 相对误差 $\epsilon = 0.001$，失败概率 $\delta = 0.01$

**解决方案**:
```
w = ⌈e/0.001⌉ = 2719
d = ⌈ln(1/0.01)⌉ = 5
空间 = 2719 × 5 × 4 bytes ≈ 54 KB
```
对比：精确统计需要存储 $2^{32}$ 个计数器 = 16 GB

**示例 2**: HyperLogLog基数估计

**问题描述**:
统计网站独立访客数（UV）。

**算法要点**:
- 使用随机哈希将元素映射为二进制串
- 记录最长前导零的个数
- 估计公式: $\hat{n} \approx \alpha_m \cdot m^2 / \sum_{j=1}^{m} 2^{-R_j}$

**结果**: 使用仅 1.5 KB 空间，相对误差约 2%。

### 5.2 代码实现

**语言**: Python

```python
import hashlib
import math
from typing import List, Tuple
import random


class CountMinSketch:
    """
    Count-Min Sketch for frequency estimation
    """
    def __init__(self, epsilon: float = 0.001, delta: float = 0.01):
        self.epsilon = epsilon
        self.delta = delta
        self.w = int(math.ceil(math.e / epsilon))
        self.d = int(math.ceil(math.log(1.0 / delta)))
        
        # Initialize counters
        self.C = [[0] * self.w for _ in range(self.d)]
        
        # Initialize hash functions (using pairwise independent family)
        self.hash_params = []
        for _ in range(self.d):
            a = random.randint(1, 2**31 - 1)
            b = random.randint(0, 2**31 - 1)
            self.hash_params.append((a, b))
    
    def _hash(self, x: str, j: int) -> int:
        """Pairwise independent hash function"""
        a, b = self.hash_params[j]
        # Use Python's string hash as base
        x_int = hash(x) & 0x7FFFFFFF
        return ((a * x_int + b) % (2**31 - 1)) % self.w
    
    def update(self, x: str, count: int = 1):
        """Process element x with count (default 1)"""
        for j in range(self.d):
            idx = self._hash(x, j)
            self.C[j][idx] += count
    
    def estimate(self, x: str) -> int:
        """Estimate frequency of x"""
        return min(self.C[j][self._hash(x, j)] for j in range(self.d))
    
    def merge(self, other: 'CountMinSketch') -> 'CountMinSketch':
        """Merge two sketches with same parameters"""
        if self.w != other.w or self.d != other.d:
            raise ValueError("Cannot merge sketches with different parameters")
        
        result = CountMinSketch(self.epsilon, self.delta)
        for j in range(self.d):
            for k in range(self.w):
                result.C[j][k] = self.C[j][k] + other.C[j][k]
        return result


class HyperLogLog:
    """
    HyperLogLog for cardinality estimation
    """
    def __init__(self, p: int = 14):
        """
        p: precision parameter (number of registers = 2^p)
        Standard error = 1.04 / sqrt(2^p)
        """
        self.p = p
        self.m = 1 << p  # 2^p
        self.registers = [0] * self.m
        self.alpha = self._get_alpha()
    
    def _get_alpha(self) -> float:
        """Bias correction constant"""
        if self.m >= 128:
            return 0.7213 / (1 + 1.079 / self.m)
        elif self.m == 64:
            return 0.709
        elif self.m == 32:
            return 0.697
        elif self.m == 16:
            return 0.673
        else:
            return 0.5
    
    def _hash(self, x: str) -> int:
        """64-bit hash"""
        h = hashlib.md5(x.encode()).hexdigest()
        return int(h[:16], 16)
    
    def _count_leading_zeros(self, value: int, bits: int = 64) -> int:
        """Count leading zeros in binary representation"""
        if value == 0:
            return bits
        return bits - value.bit_length()
    
    def add(self, x: str):
        """Add element to the set"""
        x_hash = self._hash(x)
        
        # First p bits determine register index
        idx = x_hash & (self.m - 1)
        
        # Remaining bits used for leading zeros
        w = x_hash >> self.p
        leading_zeros = self._count_leading_zeros(w, 64 - self.p) + 1
        
        self.registers[idx] = max(self.registers[idx], leading_zeros)
    
    def count(self) -> int:
        """Estimate cardinality"""
        # Harmonic mean of 2^(-register)
        Z = sum(2**(-r) for r in self.registers)
        E = self.alpha * self.m * self.m / Z
        
        # Small range correction
        if E <= 2.5 * self.m:
            V = self.registers.count(0)
            if V != 0:
                E = self.m * math.log(self.m / V)
        
        # Large range correction (for 64-bit hashes)
        elif E > (1/30) * (1 << 32):
            E = -(1 << 32) * math.log(1 - E / (1 << 32))
        
        return int(E)


# 测试
if __name__ == "__main__":
    print("=" * 50)
    print("Count-Min Sketch 测试")
    print("=" * 50)
    
    cms = CountMinSketch(epsilon=0.1, delta=0.01)
    
    # 模拟数据流
    stream = ["apple"] * 100 + ["banana"] * 50 + ["cherry"] * 25
    for item in stream:
        cms.update(item)
    
    print(f"真实频率: apple=100, banana=50, cherry=25")
    print(f"估计频率: apple={cms.estimate('apple')}, "
          f"banana={cms.estimate('banana')}, cherry={cms.estimate('cherry')}")
    
    print("\n" + "=" * 50)
    print("HyperLogLog 测试")
    print("=" * 50)
    
    hll = HyperLogLog(p=10)  # 1024 registers
    
    # 添加 10000 个独立元素
    unique_elements = set()
    for i in range(10000):
        elem = f"user_{i}"
        hll.add(elem)
        unique_elements.add(elem)
    
    # 添加一些重复
    for i in range(5000):
        hll.add(f"user_{i % 1000}")
    
    estimated = hll.count()
    actual = 10000
    error = abs(estimated - actual) / actual * 100
    
    print(f"真实基数: {actual}")
    print(f"估计基数: {estimated}")
    print(f"相对误差: {error:.2f}%")
    print(f"理论标准误差: {1.04 / math.sqrt(1024) * 100:.2f}%")
```

**代码说明**:
- `CountMinSketch`: 实现频率估计，支持更新和查询
- `HyperLogLog`: 实现基数估计，使用寄存器数组记录最长前导零
- 哈希函数使用通用哈希族保证理论性质

### 5.3 反例

**常见错误1**: 使用CM Sketch估计低频元素
```python
# 错误：期望准确估计稀有元素
cms = CountMinSketch(epsilon=0.1, delta=0.01)
# 对于频率为1的元素，估计误差可能达到 ε||f||₁
```
**错误原因**: CM Sketch对高频元素效果好，低频元素相对误差大
**正确做法**: 低频元素需要其他技术（如 Count Sketch）

**常见错误2**: 忽略哈希函数的通用性要求
```python
# 错误：使用简单取模
hash_val = x % w  # 非通用哈希
```
**错误原因**: 通用哈希保证碰撞概率上界
**正确做法**: 使用 pairwise independent 哈希族

## 六、习题

### 6.1 理解题 (L1-L2)

1. **空间复杂度计算** [难度⭐]
   
   配置Count-Min Sketch处理Universe大小为 $10^6$、期望相对误差1%、失败概率1%的流。计算所需空间。
   
   <details>
   <summary>解答</summary>
   
   $w = \lceil e/0.01 \rceil = 272$
   
   $d = \lceil \ln(1/0.01) \rceil = 5$
   
   假设计数器需要4字节：
   空间 = $272 \times 5 \times 4 = 5440$ 字节 ≈ 5.3 KB
   
   </details>

2. **近似保证解释** [难度⭐⭐]
   
   解释Count-Min Sketch为什么是单边估计（$f_i \leq \hat{f}_i$），以及为什么估计值不会超过 $f_i + \epsilon \|f\|_1$。
   
   <details>
   <summary>解答</summary>
   
   单边估计原因：哈希碰撞只会增加计数器的值，不会减少。因此估计值至少为真实值。
   
   上界原因：误差来自哈希碰撞。每个其他元素以概率 $1/w$ 碰撞。期望误差为 $(\|f\|_1 - f_i)/w \leq \epsilon \|f\|_1 / e$。通过马尔可夫不等式和最小值操作，高概率保证误差不超过 $\epsilon \|f\|_1$。
   
   </details>

### 6.2 应用题 (L3)

3. **频繁项检测** [难度⭐⭐⭐]
   
   设计算法使用Count-Min Sketch检测"Heavy Hitters"（频率超过 $\phi \|f\|_1$ 的元素）。
   
   <details>
   <summary>解答</summary>
   
   **算法**:
   1. 使用CM Sketch跟踪所有元素频率
   2. 维护候选集 $S$（大小 $O(1/\phi)$）
   3. 对每条记录 $(x, c)$：
      - 更新CM Sketch
      - 若 $\hat{f}_x \geq \phi \cdot \text{total}$，加入候选集
   4. 定期清理候选集中频率估计低于阈值元素
   
   **保证**: 所有heavy hitters会被报告，假阳性频率不超过 $(\phi - \epsilon)\|f\|_1$
   
   </details>

### 6.3 证明题 (L4-L5)

4. **AMS Sketch方差分析** [难度⭐⭐⭐⭐]
   
   AMS Sketch用于估计二阶矩 $F_2 = \sum_i f_i^2$。证明其估计量的方差为 $O(F_2^2)$，并说明如何通过平均 $k$ 个独立估计将误差降至 $O(1/\sqrt{k}}$)。
   
   <details>
   <summary>解答</summary>
   
   **AMS Sketch**: 维护随机变量 $Z = (\sum_i f_i \cdot r_i)^2$，其中 $r_i \in \{-1, +1\}$ 是4-wise independent随机变量。
   
   **期望**: 
   $$E[Z] = E[\sum_{i,j} f_i f_j r_i r_j] = \sum_i f_i^2 E[r_i^2] + \sum_{i \neq j} f_i f_j E[r_i r_j] = F_2$$
   
   **方差**: 
   $$Var(Z) = E[Z^2] - F_2^2$$
   展开 $Z^2$ 并利用4-wise独立性，可得 $Var(Z) \leq 2F_2^2$。
   
   **平均**: 取 $k$ 个独立估计的平均，方差变为 $2F_2^2/k$，标准差为 $O(F_2/\sqrt{k})$。
   
   </details>

5. **Distinct Elements下界** [难度⭐⭐⭐⭐⭐]
   
   证明任何确定性算法精确计算distinct elements需要 $\Omega(n)$ 空间。
   
   <details>
   <summary>解答</summary>
   
   **归约**: 从集合不交问题（DISJ）归约。
   
   Alice持有 $A \subseteq [n]$，Bob持有 $B \subseteq [n]$。他们想知道 $A \cap B = \emptyset$。
   
   **构造流**: Alice先输入 $A$ 的所有元素，Bob再输入 $B$ 的所有元素。
   
   **关键**: $D = |A \cup B| = |A| + |B| - |A \cap B|$。
   
   若算法能在 $o(n)$ 空间精确计算 $D$，则 Alice 可以将内部状态发送给 Bob（$o(n)$ 比特），Bob可计算 $|A \cap B|$，解决DISJ。
   
   但DISJ的通信复杂度为 $\Omega(n)$，矛盾。因此需要 $\Omega(n)$ 空间。∎
   
   </details>

**案例**: Google的Distinct Counting系统

**背景**:
Google需要统计搜索查询的独立用户数（UV），每日处理数万亿查询。

**技术方案**:
- 使用改进的HyperLogLog++算法
- 64位哈希支持（原始HyperLogLog为32位）
- 稀疏表示优化小基数场景
- 偏差修正提高精度

**效果**:
- 内存占用减少约80%
- 相对误差控制在2%以内
- 支持实时聚合和并行处理

## 八、多维对比

### 8.1 数据结构对比

| 数据结构 | 空间复杂度 | 支持查询 | 误差类型 | 适用场景 |
|----------|------------|----------|----------|----------|
| Count-Min Sketch | $O(\epsilon^{-1}\log \delta^{-1})$ | 点查询 | 单边 | 频率估计 |
| Count Sketch | $O(\epsilon^{-2}\log \delta^{-1})$ | 点查询, 内积 | 双边 | 低频准确 |
| AMS Sketch | $O(\epsilon^{-2}\log \delta^{-1})$ | $F_2$估计 | 双边 | 矩估计 |
| HyperLogLog | $O(\epsilon^{-2}\log\log n)$ | 基数 | 双边 | Distinct Count |
| Bloom Filter | $O(n \log(1/\epsilon))$ | 成员查询 | 单边 | 集合测试 |
| Quantile Sketch | $O(\epsilon^{-1}\log(\epsilon n))$ | 分位数 | 双边 | 分布估计 |

### 8.2 算法模型对比

| 模型 | 空间需求 | 处理能力 | 典型应用 |
|------|----------|----------|----------|
| Cash Register | $O(\text{polylog}(n))$ | 频率、基数 | 网络统计 |
| Turnstile | $O(\text{polylog}(n))$ | 线性草图 | 数据库 |
| Sliding Window | $O(W)$ 或 $O(\epsilon^{-1}\log W)$ | 最近数据 | 实时监控 |
| Distributed | $O(\text{polylog}(n))$ + 通信 | 全局聚合 | 传感器网络 |

### 8.3 决策建议

**选择Sketch结构的流程**:
```
需要回答什么问题？
├── 频率估计
│   ├── 只关心高频 → Count-Min Sketch
│   └── 需要低频准确 → Count Sketch
├── Distinct Count → HyperLogLog
├── 成员测试 → Bloom Filter
├── 分位数 → T-Digest / KLL Sketch
└── 矩估计 / 内积 → AMS Sketch
```
