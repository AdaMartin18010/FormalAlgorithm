## 一、核心概念

### 1.1 定义

**正式定义**:
**共识问题（Consensus Problem）**: $n$ 个进程在异步网络中，各自提出一个值，最终需满足：
1. **一致同意（Agreement）**: 所有正确进程决定相同值
2. **有效性（Validity）**: 若所有正确进程提出相同值 $v$，则决定值必为 $v$
3. **终止性（Termination）**: 所有正确进程最终必须决定

**FLP不可能性结果**:
在异步系统中，即使只有一个进程可能故障，也不存在确定性的共识算法。

**Paxos算法**:
- **角色**: Proposer（提议者）、Acceptor（接受者）、Learner（学习者）
- **阶段**: Prepare阶段（承诺）和 Accept阶段（接受）
- **多数派**: 需要 $> n/2$ 接受者响应

**Raft算法**:
- **角色**: Leader（领导者）、Follower（跟随者）、Candidate（候选者）
- **机制**: 领导者选举 + 日志复制
- **任期（Term）**: 逻辑时钟，单调递增

**直观解释**:
共识算法就像"一群人决定晚餐吃什么"。Paxos像"两阶段提交"的民主投票：先询问大家是否愿意接受某提议（Prepare），再正式提议（Accept）。Raft像"选举班长"：先选出Leader，然后Leader统一协调决策。如果Leader挂了，重新选举。

**关键要点**:
- **FLP权衡**: 实际系统选择牺牲终止性（超时）或安全性的边界情况
- **多数派机制**: 保证两个多数派必有交集
- **Leader选举**: 避免活锁，简化正常情况路径
- **日志匹配**: 保证状态机复制的正确性

### 1.2 属性

| 属性 | 描述 | 备注 |
|------|------|------|
| 安全性 | 正确性永不违反 | 即使网络分区 |
| 活性 | 最终能做出决定 | 需网络稳定假设 |
| 容错性 | 容忍故障节点数 | 通常 $f < n/2$ |
| 性能 | 正常情况延迟 | Leader-based优化 |

**性质总结**:
1. **Paxos的安全性**: 由提议编号的单调性保证
2. **Raft的选举安全**: 每个任期最多一个Leader
3. **日志匹配**: 相同索引和任期的日志条目内容相同
4. **状态机安全**: 若某节点应用了某日志，其他节点不会在相同位置应用不同日志

### 1.3 变体

**Multi-Paxos**:
- 定义: 优化连续多个值的共识
- 与Basic Paxos的区别: Leader稳定时跳过Prepare阶段
- 适用场景: 高吞吐复制

**Raft (Ongaro & Ousterhout)**:
- 定义: 可理解的共识算法
- 与Paxos的区别: 更强的Leader角色，更易理解
- 适用场景: 教学、实际系统（etcd、Consul）

**Flexible Paxos**:
- 定义: 放宽Quorum约束
- 与传统Paxos的区别: Prepare和Accept可用不同Quorum
- 适用场景: 广域网复制

## 三、形式化证明

### 3.1 核心定理

**定理 1 (Paxos的安全性)**: 若一个值 $v$ 被某个Acceptor接受，则任何被多数派接受的值必为 $v$。

**证明**:

**关键观察**: 任意两个多数派必有交集（鸽巢原理）。

**归纳于提议编号**:

基础: 第一个被接受的值，显然成立。

归纳: 假设对所有编号 $< n$ 成立，考虑编号 $n$。

**Prepare阶段**: Proposer在提议编号 $n$ 前，已得到多数派承诺不接受更低编号的提议。

**Accept阶段**: 
- 若Proposer从多数派获得已接受值，选择最高编号的值 $v$ 提议
- 若无已接受值，可提议任意值

由于任何先前接受的值必在某个Prepare响应中（多数派交集），Proposer必能获知。

因此，若值 $v$ 已被接受，后续提议者必获知并提议相同值。

∎

**证明要点分析**:
1. 多数派交集保证信息传递
2. 提议编号的全序保证决策单调性
3. 两阶段分离承诺和决策

**定理 2 (Raft的安全性)**: 若Leader在任期 $T$ 提交了日志条目，则所有后续任期的Leader必包含该条目。

**证明**:

Leader选举约束: 候选者必须获得多数派投票，投票前检查候选者日志至少与投票者一样新。

**日志比较**: 
- 先比较最后日志任期，任期高者更新
- 任期相同，比较日志长度

若Leader $L_1$ 在任期 $T$ 提交了条目 $e$（已复制到多数派），则任何获得多数派的候选者 $C$ 必从包含 $e$ 的节点获得至少一票。

由于 $C$ 的日志至少与投票者一样新，$C$ 必包含 $e$。

∎

### 3.2 辅助引理

**引理 1 (多数派交集)**: 在 $n$ 个节点的系统中，任意两个大小 $> n/2$ 的集合必有非空交集。

*证明*: $|A| + |B| > n$，由鸽巢原理，$|A \cap B| = |A| + |B| - |A \cup B| > n - n = 0$。∎

**引理 2 (Raft的选举限制)**: 若候选者 $C$ 在任期 $T$ 当选，则 $C$ 的日志包含所有在任期 $< T$ 已提交的条目。

*证明*: 由投票检查机制和多数派交集。∎

## 四、算法/方法详解

### 4.1 算法描述

**Basic Paxos**:
```
角色: Proposer p, Acceptor集合 A, Learner集合 L

Phase 1a (Prepare):
    p选择新提议编号 n，发送 Prepare(n) 给多数派

Phase 1b (Promise):
    若 a ∈ A 未承诺更高编号:
        a承诺不接受编号 < n 的提议
        回复 Promise(n, 已接受最高编号的值)

Phase 2a (Accept):
    若 p 收到多数派 Promise:
        若有返回的已接受值，选择最高编号的值 v
        否则选择自己的提议值 v
        发送 Accept(n, v) 给多数派

Phase 2b (Accepted):
    若 a ∈ A 已承诺编号 n:
        接受值 v
        回复 Accepted(n, v)
        通知 Learners

Learner:
    收到多数派对相同 (n, v) 的 Accepted:
        学习值 v
```

**Raft领导者选举**:
```
Follower:
    选举超时（随机150-300ms）:
        转为Candidate，term++，投票给自己
        向所有节点发送 RequestVote RPC

Candidate:
    收到多数派投票:
        成为Leader，发送心跳
    收到更高term的AppendEntries:
        转为Follower
    选举超时:
        重新选举（term++）

RequestVote处理:
    若 term < currentTerm:
        拒绝
    若 term > currentTerm:
        更新term，转为Follower
    若 未投票且候选者日志至少一样新:
        投票

日志比较:
    (lastLogTerm, lastLogIndex) 字典序
```

**Raft日志复制**:
```
Leader:
    收到客户端命令:
        追加到本地日志
        并行发送 AppendEntries 给所有Follower
    收到多数派成功响应:
        提交条目，应用到状态机
        后续AppendEntries通知Follower提交

Follower处理AppendEntries:
    若 term < currentTerm:
        拒绝
    若 prevLogIndex/Term 不匹配:
        拒绝，Leader递减nextIndex重试
    删除冲突条目，追加新条目
    若 leaderCommit > commitIndex:
        commitIndex = min(leaderCommit, 最后新条目索引)
```

**流程图**:
```
Paxos:                    Raft:
Prepare → Promise         Follower → Candidate → Leader
   ↓                          ↓         ↑
Accept → Accepted         Log Replication
   ↓                          ↓
Learn                     Commit → Apply
```

### 4.2 正确性分析

**Paxos正确性**:
- **一致性**: 由多数派交集和提议编号保证
- **非平凡性**: 只有被提议的值可被接受
- **终止性**: 无竞争时两轮完成（实际系统需超时重试）

**Raft正确性**:
- **选举安全**: 每个任期最多一个Leader（多数派投票）
- **日志匹配**: 相同位置相同任期的日志内容相同
- **Leader完备性**: Leader包含所有已提交日志
- **状态机安全**: 相同位置的日志被一致应用

### 4.3 复杂度分析

**Basic Paxos**:
- 延迟: 2轮RPC（Prepare + Accept）
- 消息: $O(n)$ 每提议
- 高吞吐优化: Multi-Paxos减少为1轮（Leader稳定）

**Raft**:
- 选举: 最坏 $O(n^2)$ 消息（多次分裂投票）
- 正常操作: $O(n)$ 消息每条目
- 复制延迟: 1轮RPC（Leader → Follower → Leader）

## 五、示例与实例

### 5.1 标准示例

**示例 1: Raft领导者选举场景**

**初始状态**: 5个节点（A, B, C, D, E），无Leader

**场景**:
1. A、B同时超时，分别变为Candidate，term=1
2. A向所有节点发送RequestVote
3. C收到A的请求：未投票，A日志新，投票给A
4. D收到B的请求：未投票，B日志新，投票给B
5. A收到C的投票：获得3票（自己+C），成为Leader
6. B收到A的心跳（term=1）：转为Follower

**结果**: A成为term=1的Leader

**示例 2: 日志复制冲突解决**

**场景**: Leader（term=3）发送AppendEntries
- prevLogIndex=5, prevLogTerm=3
- entries=[(6,3,"cmd1")]

**Follower响应**:
- 若Follower在index=5处term=3: 成功追加
- 若Follower在index=5处term=2（落后）: 失败，Leader递减nextIndex重试
- 若Follower在index=5处term=4（更超前）: 失败，Leader转为Follower

**冲突解决**: Leader递减nextIndex直到匹配，删除Follower冲突条目，发送缺失条目

from typing import Dict, List, Optional, Set
from dataclasses import dataclass, field
import random
import threading
import time

class NodeState(Enum):
    FOLLOWER = auto()
    CANDIDATE = auto()
    LEADER = auto()

@dataclass
class LogEntry:
    term: int
    command: str
    index: int

@dataclass
class RaftNode:
    node_id: str
    peers: List[str]
    
    # 持久状态
    current_term: int = 0
    voted_for: Optional[str] = None
    log: List[LogEntry] = field(default_factory=list)
    
    # 易失状态
    state: NodeState = NodeState.FOLLOWER
    commit_index: int = 0
    last_applied: int = 0
    
    # Leader状态
    next_index: Dict[str, int] = field(default_factory=dict)
    match_index: Dict[str, int] = field(default_factory=dict)
    
    # 选举定时器
    election_timeout: float = 0.0
    
    def __post_init__(self):
        self.reset_election_timeout()
        self.log.append(LogEntry(0, "", 0))  # 哨兵条目
    
    def reset_election_timeout(self):
        """随机选举超时 150-300ms"""
        self.election_timeout = time.time() + 0.15 + random.random() * 0.15
    
    def start_election(self):
        """开始选举"""
        self.state = NodeState.CANDIDATE
        self.current_term += 1
        self.voted_for = self.node_id
        self.reset_election_timeout()
        
        votes = {self.node_id}  # 自己投票
        
        # 发送RequestVote
        last_log = self.log[-1] if self.log else LogEntry(0, "", 0)
        for peer in self.peers:
            # 模拟RPC发送
            pass
        
        return votes
    
    def request_vote(self, candidate_id: str, term: int,
                    last_log_index: int, last_log_term: int) -> bool:
        """处理投票请求"""
        if term > self.current_term:
            self.current_term = term
            self.state = NodeState.FOLLOWER
            self.voted_for = None
        
        if term < self.current_term:
            return False
        
        # 检查日志至少一样新
        my_last = self.log[-1] if self.log else LogEntry(0, "", 0)
        log_ok = (last_log_term > my_last.term or 
                  (last_log_term == my_last.term and 
                   last_log_index >= my_last.index))
        
        if (self.voted_for is None or self.voted_for == candidate_id) and log_ok:
            self.voted_for = candidate_id
            self.reset_election_timeout()
            return True
        
        return False
    
    def append_entries(self, leader_id: str, term: int,
                      prev_log_index: int, prev_log_term: int,
                      entries: List[LogEntry], leader_commit: int) -> bool:
        """处理日志复制请求"""
        if term > self.current_term:
            self.current_term = term
            self.state = NodeState.FOLLOWER
            self.voted_for = None
        
        if term < self.current_term:
            return False
        
        # 有效Leader心跳
        self.state = NodeState.FOLLOWER
        self.reset_election_timeout()
        
        # 检查日志匹配
        if prev_log_index >= len(self.log):
            return False
        
        if self.log[prev_log_index].term != prev_log_term:
            return False
        
        # 删除冲突条目，追加新条目
        self.log = self.log[:prev_log_index + 1]
        self.log.extend(entries)
        
        # 更新commitIndex
        if leader_commit > self.commit_index:
            self.commit_index = min(leader_commit, len(self.log) - 1)
        
        return True
    
    def become_leader(self):
        """成为Leader"""
        self.state = NodeState.LEADER
        for peer in self.peers:
            self.next_index[peer] = len(self.log)
            self.match_index[peer] = 0
    
    def replicate_log(self, peer: str) -> bool:
        """向指定节点复制日志"""
        if self.state != NodeState.LEADER:
            return False
        
        next_idx = self.next_index[peer]
        prev_log = self.log[next_idx - 1] if next_idx > 0 else LogEntry(0, "", 0)
        entries = self.log[next_idx:]
        
        # 模拟发送AppendEntries
        # 若成功: self.next_index[peer] += len(entries), self.match_index[peer] = ...
        # 若失败: self.next_index[peer] -= 1
        
        return True

### 5.3 反例

**常见错误**:

**错误1: 忽视FLP结果**
```python
# 错误：期望异步网络中确定性共识
def consensus(values):
    # ... 实现 ...
    return agreed_value  # 可能永远阻塞！
```
**错误原因**: 异步网络中无法同时保证安全性和活性
**正确做法**: 使用超时和随机化，接受概率性终止

**错误2: 日志截断破坏安全性**
```python
# 错误：随意截断日志
if log_size > MAX_SIZE:
    log = log[-100:]  # 截断早期日志！
```
**错误原因**: 可能删除已提交但未应用的日志
**正确做法**: 仅在快照后截断已提交的日志

## 六、习题

### 6.1 理解题

1. **解释**: 为什么Raft的日志复制机制能保证状态机安全（State Machine Safety）？

   <details>
   <summary>解答</summary>
   
   状态机安全: 若某服务器在特定索引应用了日志到状态机，其他服务器不会在同一索引应用不同日志。
   
   **保证机制**:
   
   1. **Leader完备性**: Leader包含所有已提交的日志
      - 选举时投票检查确保新Leader日志足够新
   
   2. **日志匹配**: 若两日志在相同索引有相同任期，则它们在该索引前完全一致
      - 通过AppendEntries的prevLog检查保证
   
   3. **提交条件**: Leader只提交当前任期的日志条目
      - 防止旧Leader的日志被错误提交
   
   4. **冲突解决**: Follower删除与Leader冲突的所有后续日志
      - 保证所有节点最终与Leader一致
   
   因此，所有节点在相同位置看到相同日志，状态机应用结果一致。
   </details>

### 6.2 应用题

1. **设计**: 设计一个Raft集群的成员变更协议，允许在线添加/删除节点。

   <details>
   <summary>解答</summary>
   
   联合共识（Joint Consensus）方法：
   
   1. **变更阶段**:
      - Leader将新配置作为日志条目提交
      - 新旧配置同时生效（联合配置）
      - 需要新旧配置各自的多数派同意
   
   2. **过渡**:
      - 新节点作为非投票成员加入，追赶日志
      - 确认同步后转为投票成员
   
   3. **完成**:
      - 提交新配置条目
      - 仅新配置生效
   
   ```python
   def joint_consensus(old_config, new_config):
       # 阶段1: 联合配置
       joint_config = old_config.union(new_config)
       replicate_config_change(joint_config)
       
       # 阶段2: 新配置
       replicate_config_change(new_config)
   ```
   
   安全保证：
   - 任何两个联合配置的多数派必有交集
   - 防止脑裂
   </details>

### 6.3 证明题

1. **证明**: 在Raft中，若某日志条目在某任期被提交，则该条目会出现在所有更高任期的Leader的日志中。

   <details>
   <summary>解答</summary>
   
   **证明**:
   
   设条目 $e$ 在任期 $T$ 被提交。
   
   **关键事实**: $e$ 已复制到 $T$ 任期的多数派。
   
   **选举论证**: 
   - 任何任期 $T' > T$ 的新Leader必须获得多数派投票
   - 任何多数派与 $T$ 的多数派必有交集
   - 交集节点已包含 $e$
   
   **投票检查**: 
   - 候选者请求投票时，节点检查候选者日志是否至少一样新
   - 包含 $e$ 的节点只投票给同样包含 $e$ 的候选者
   
   **结论**: 
   - 新Leader必从包含 $e$ 的节点获得至少一票
   - 因此新Leader的日志至少与投票者一样新，包含 $e$
   
   ∎
   </details>

### 8.2 决策建议

**何时选择Raft**:
- 需要可理解、可维护的代码
- Leader-based工作负载
- 教学或新系统开发

**何时选择EPaxos**:
- 高冲突工作负载
- 广域网部署
- 需要最大可用性
