# 案例：归并排序在外部排序与大数据处理中的应用

## 背景

当数据量远超内存容量时（如 100GB 日志文件排序），无法将全部数据载入内存进行快速排序。**外部排序（External Sorting）**利用归并排序的思想，将数据分块读入内存排序，然后多路归并，成为大数据处理的标准方法。

## 问题定义

- **输入**：$N$ 个记录，总大小 >> 内存容量 $M$
- **约束**：每次只能将 $B$ 个记录读入内存（$B \ll N$）
- **目标**：完成全排序，最小化磁盘 I/O 次数

## 外部排序算法

### 阶段 1：生成有序段（Runs）

1. 每次读入 $M$ 个记录到内存
2. 内存排序（快速排序或归并排序）
3. 写回磁盘为一个有序段

段数：$\lceil N/M \rceil$

### 阶段 2：k-路归并

1. 从 $k$ 个段中各读入一个块到内存
2. 维护最小堆，每次输出最小元素
3. 当某段的缓冲区空时，从磁盘读入下一块

**I/O 次数**：$O(N/B \cdot \log_{M/B} (N/M))$

## 实际系统

### Linux sort 命令

```bash
sort -S 1G huge_file.txt > sorted.txt
```

- `-S` 指定内存缓冲区大小
- 内部使用外部归并排序

### Hadoop MapReduce

- **Map**：输出键值对，框架按 key 分区并排序
- **Shuffle**：各 reducer 拉取属于自己分区的有序数据
- **Reduce**：处理有序输入，可高效进行分组聚合

### 数据库索引构建

- B+ 树索引的批量加载使用外部排序
- PostgreSQL 的 `CREATE INDEX` 对大数据表使用外部归并

## 与快速排序的对比

| 场景 | 快速排序 | 归并排序 |
|------|---------|---------|
| 内存内排序 | 通常更快（缓存友好）| 稳定，适合链表 |
| 外部排序 | ❌ 不适合 | ✅ 天然适合顺序 I/O |
| 并行排序 | 划分困难 | 易并行（多路归并）|
| 链表排序 | $O(n \log n)$ 需要额外空间 | $O(n \log n)$，$O(1)$ 额外 |

## 效果评估

- 对 100GB 数据、1GB 内存，外部归并排序约需 30-60 分钟
- I/O 为瓶颈，SSD 可比 HDD 快 3-5 倍
- 现代数据库使用 SSD + 大内存（64GB+），可将 1TB 排序降至分钟级
