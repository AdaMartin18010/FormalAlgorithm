# 案例：跳表在数据库索引与分布式系统中的应用

## 背景

跳表（Skip List）是 William Pugh 提出的概率性数据结构，以期望 $O(\log n)$ 的时间支持搜索、插入和删除，实现比平衡树简单得多。
Redis 的有序集合和 LevelDB 的 MemTable 均使用跳表作为核心索引结构。

## 问题定义

- **输入**：有序元素集合，支持动态增删
- **操作**：搜索、插入、删除、范围查询
- **目标**：期望 $O(\log n)$ 时间，实现简单

## 跳表结构

### 层级设计

- 每层是有序链表
- 第 0 层包含所有元素
- 第 $i$ 层元素以概率 $p$（通常 $1/2$）出现在第 $i+1$ 层
- 期望层数：$O(\log n)$

### 搜索

从最高层开始，向右或向下移动：

```
while curr.next[level] < target:
    curr = curr.next[level]
if curr.next[level] == target: found
else: level--, continue
```

## 实际应用

### Redis 有序集合（Sorted Set）

- `ZADD`、`ZREM`、`ZRANGEBYSCORE` 底层使用跳表 + 哈希表
- 跳表按分数排序，支持范围查询
- 哈希表提供 O(1) 的按成员查找
- 百万级元素，操作 $< 1$ms

### LevelDB / RocksDB

- MemTable 使用跳表存储内存中的键值对
- 按 key 有序，支持迭代器遍历
- 持久化时转换为 SSTable

### 分布式系统

- 一致性哈希的虚拟节点管理
- 分布式跳表用于范围分区（如 Bigtable）

### 替代平衡树

| 场景 | 跳表 | 红黑树/AVL |
|------|------|-----------|
| 实现复杂度 | 简单 ✅ | 复杂 |
| 期望性能 | $O(\log n)$ | $O(\log n)$ 确定性 |
| 范围查询 | ✅ 天然支持 | 需中序遍历 |
| 并发控制 | ✅ 锁粒度细 | 较难 |
| 空间开销 | 较大 | 较小 |

## 效果评估

- Redis 有序集合：百万元素，插入/删除 $< 100\mu s$
- LevelDB MemTable：跳表性能接近平衡树，实现更简单
- 并发跳表（如 Java `ConcurrentSkipListMap`）在多核上扩展性好
