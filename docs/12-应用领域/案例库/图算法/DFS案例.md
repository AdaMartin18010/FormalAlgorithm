# 案例：DFS在代码依赖分析与死代码检测中的应用

## 背景

现代软件项目包含大量模块、函数和类，它们之间存在复杂的调用关系。随着项目演进，部分代码不再被任何入口调用（死代码），需要清理以减少维护负担。DFS（深度优先搜索）是遍历调用图、分析依赖关系和检测死代码的标准算法。

## 问题定义

- **调用图**：有向图 $G=(V,E)$，顶点为函数/模块，边为调用关系（$A \to B$ 表示 A 调用 B）
- **入口点**：程序启动时调用的函数（main、回调注册点等）
- **目标**：找出从入口点不可达的所有顶点（死代码）

## DFS 的应用

### 死代码检测

1. 从所有入口点开始 DFS
2. 标记访问过的所有函数
3. 未被标记的函数即为死代码

### 循环依赖检测

DFS 遍历过程中记录访问栈：

- 若遇到已在栈中的节点，发现环
- 用于检测模块间的循环引用（如 Python、Node.js 的导入循环）

### 拓扑排序

- DFS 完成后按**完成时间逆序**输出节点
- 得到合法的编译/构建顺序
- TypeScript、Java 编译器使用类似机制

## 与 BFS 的对比

| 场景 | DFS | BFS |
|------|-----|-----|
| 死代码检测 | ✅ 栈空间小，适合深调用链 | ✅ 同样可行 |
| 循环检测 | ✅ 利用递归栈天然检测 | ❌ 需额外记录路径 |
| 最短调用链 | ❌ 不保证最短 | ✅ 天然最短 |
| 内存占用 | $O(d)$，$d$ 为深度 | $O(b^d)$，较大 |

## 实际工具

| 工具 | 功能 | DFS 应用 |
|------|------|---------|
| Webpack | JS 打包 | Tree Shaking（标记未使用导出）|
| Rollup | ES 模块打包 | 基于入口的 DFS 剪枝 |
| TypeScript | 类型检查 | 模块依赖图遍历 |
| GCC/Clang | C/C++ 编译 | 死函数消除（LTO）|
| Pyflakes | Python 静态分析 | 未使用导入检测 |

## 效果评估

- 对 100,000 个函数的调用图，DFS 标记 < 100ms
- Tree Shaking 可将前端包体积减少 30-70%
- 死代码检测在 CI 流程中自动执行，防止技术债务累积
