# 案例：字典树在搜索引擎自动补全与输入法中的应用

## 背景

搜索引擎、IDE 和输入法需要根据用户输入的前缀，快速推荐可能的补全选项。字典树（Trie，前缀树）以 $O(L)$ 的时间完成前缀匹配（$L$ 为前缀长度），是这类场景的标准数据结构。

## 问题定义

- **词典**：大量字符串集合（搜索关键词、API 名称、词汇表）
- **查询**：前缀 $p$
- **目标**：找到所有以 $p$ 为前缀的字符串

## 字典树结构

- 每个节点代表一个字符
- 从根到叶子的路径构成一个单词
- 节点标记是否为单词结尾

### 操作复杂度

| 操作 | 复杂度 |
|------|--------|
| 插入 | $O(L)$ |
| 前缀搜索 | $O(L + K)$，$K$ 为结果数 |
| 精确查找 | $O(L)$ |
| 删除 | $O(L)$ |

## 实际应用

### 搜索引擎自动补全

- 用户输入"algor"，推荐"algorithm"、"algorithms"、"algorithmic"
- Trie 存储数十亿搜索历史的高频前缀
- 结合频率排序，推荐最热门的补全

### 输入法（拼音/日文）

- 拼音 Trie：音节 → 汉字候选
- 用户输入 "zhongguo"，匹配 "中国"、"中过"、"种过"等
- 结合语言模型排序

### IDE 代码补全

- 类名、方法名、变量名的前缀匹配
- 大型代码库中，Trie 索引符号表
- VS Code、IntelliJ 的 IntelliSense 底层使用类似结构

### IP 路由表查找

- 最长前缀匹配（Longest Prefix Match）
- 二进制 Trie（Patricia Trie）存储路由前缀
- 路由器中硬件实现（TCAM）

### 拼写检查

- Trie 存储词典
- 前缀不匹配时提前剪枝，加速模糊匹配

## 优化变体

| 变体 | 特点 | 用途 |
|------|------|------|
| 压缩 Trie | 合并单分支节点 | 节省空间 |
| Patricia Trie | 二进制前缀树 | IP 路由、哈希替代 |
| Suffix Tree | 所有后缀的 Trie | 子串搜索 |
| Suffix Array | 后缀数组 + LCP | 更省空间的子串搜索 |
| Radix Tree | 基数树 | 分布式系统（etcd）|

## 与哈希表的对比

| 操作 | Trie | 哈希表 |
|------|------|--------|
| 精确查找 | $O(L)$ | $O(1)$ ✅ |
| 前缀搜索 | $O(L + K)$ ✅ | ❌ 需遍历全表 |
| 有序遍历 | ✅ 天然有序 | ❌ 需排序 |
| 空间 | 较大 | 较小 |

**选择建议**：

- 仅需精确查找：哈希表
- 需要前缀匹配/自动补全：Trie
- 需要有序遍历 + 前缀匹配：Trie 或有序集合

## 效果评估

- Trie 前缀查询：百万级词典，查询 < 1ms
- 搜索引擎自动补全：Trie + 频率索引，延迟 < 10ms
- 压缩 Trie（DAWG）将空间减少 50-90%
