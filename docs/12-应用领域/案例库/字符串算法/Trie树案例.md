# 案例：Trie树在搜索引擎查询建议与自动补全中的应用

## 背景

搜索引擎的查询建议（Query Suggestion）和输入法的自动补全是现代交互的核心功能。当用户输入前缀 "alg" 时，系统需要在毫秒级返回 "algorithm"、"algebra"、"algae" 等建议词。Trie 树（前缀树）是实现这一功能的最自然数据结构。

## 问题定义

- **词库**：数百万个查询词/词汇，带有频率/权重
- **输入**：用户已输入的前缀 $p$
- **输出**：以 $p$ 为前缀的 Top-K 最相关词汇
- **约束**：响应时间 < 50ms

## Trie 树的应用

### 前缀匹配

从 Trie 根节点沿前缀 $p$ 走到对应节点，然后遍历整个子树收集所有词汇：

- 时间：$O(|p| + N)$，$N$ 为匹配数
- 对热门前缀可能返回数万条结果，需进一步截断

### Top-K 优化

在每个 Trie 节点维护**子树中的热门词汇缓存**：

- 节点存储其子树中频率最高的 $K$ 个词（如 $K=10$）
- 前缀查询时直接返回缓存，无需遍历子树
- 空间换时间，查询降至 $O(|p|)$

### 压缩 Trie（Radix Tree / Patricia Tree）

合并单分支节点，减少指针开销：

- 标准 Trie：每个字符一个节点
- Radix Tree：每条边存储字符串片段
- 在路由表（IP 前缀匹配）中广泛使用

## 实际系统

### 搜索引擎（Google / Bing）

- Trie 存储数十亿查询词的历史记录
- 每个节点关联点击率、转化率等信号
- 结合机器学习模型重排 Top-K

### 输入法

- 用户个人词库用 Trie 维护
- 支持拼音到汉字的转换（多音字处理）
- 云输入法同步 Trie 结构的个性化数据

### IDE 代码补全

- 代码标识符（变量、函数、类名）存入 Trie
- 结合类型信息过滤（如只显示当前作用域内的标识符）

## 与哈希前缀树的对比

| 方法 | 构建 | 前缀查询 | Top-K | 内存 |
|------|------|---------|-------|------|
| Trie | $O(n \cdot L)$ | $O(|p| + K)$ | 天然支持 | 高 |
| 排序数组 + 二分 | $O(nL \log n)$ | $O(|p| + \log n + K)$ | 需额外结构 | 低 |
| FST（有限状态机）| $O(nL)$ | $O(|p| + K)$ | 支持 | 极低 |

**FST（Finite State Transducer）**是 Trie 的最小化版本，用于 Lucene、Elasticsearch 等系统，内存仅为 Trie 的 1/10。

## 效果评估

- 对 100 万词汇的 Trie，前缀查询 < 1ms
- Radix Tree 压缩后内存占用降低 60-80%
- 支持每秒 10 万级并发查询（配合缓存）
