## 一、核心概念

### 1.1 定义

**正式定义**:
```
倒排索引（Inverted Index）是一种从词项（term）到文档集合中
包含该词项的文档列表的映射数据结构。

形式上，设 D = {d₁, d₂, ..., dₙ} 为文档集合，
T = {t₁, t₂, ..., tₘ} 为所有不同词项的集合。

倒排索引是一个函数：
I: T → 2^D × ℕ^ℕ

其中 I(t) = ⟨(d₁, f₁), (d₂, f₂), ..., (dₖ, fₖ)⟩
- dᵢ: 包含词项 t 的文档ID
- fᵢ: 词项 t 在文档 dᵢ 中的词频（term frequency）
- 列表按文档ID升序排列
```

**直观解释**:
传统索引（正排索引）是"文档→词项"的映射，就像书的目录；
倒排索引则是"词项→文档"的映射，就像书的索引。

例如，对于文档集：
- D1: "信息检索技术"
- D2: "检索算法与数据结构"
- D3: "信息抽取技术"

倒排索引为：
- "信息" → [D1, D3]
- "检索" → [D1, D2]
- "技术" → [D1, D3]
- "算法" → [D2]
- "数据结构" → [D2]

**关键要点**:
- **词项字典（Dictionary）**: 存储所有唯一词项，通常用 B-树或哈希表实现
- **倒排列表（Posting List）**: 每个词项对应的文档列表，是索引的核心
- **位置信息（Position Info）**: 支持短语查询和邻近查询
- **文档频率（DF）**: 包含该词项的文档数量，用于排序和剪枝

### 1.2 属性

| 属性 | 描述 | 备注 |
|------|------|------|
| 时间效率 | 查询时间 O(1) 查字典 + O(k) 合并列表 | k为候选文档数 |
| 空间效率 | 通常为正排索引的 30%-50% | 经压缩后 |
| 更新成本 | 增量更新或重建，成本较高 | 通常采用定期重建策略 |
| 支持查询 | 精确匹配、布尔查询、短语查询 | 依赖位置索引 |
| 可扩展性 | 可水平分片（Sharding） | 支持PB级数据 |

**性质总结**:
1. **有序性**: 倒排列表按文档ID排序，支持高效的列表交集/并集操作
2. **稀疏性**: 大多数词项出现在少数文档中， posting list 通常很短
3. **幂律分布**: 词项频率服从 Zipf 定律，高频词项占比极少但 posting list 很长

### 1.3 变体

**位置索引（Positional Index）**:
- 定义: 在 posting 中存储词项在文档中的位置信息
- 与标准形式的区别: 标准形式只存储 (docID, tf)，位置索引存储 (docID, tf, [pos₁, pos₂, ...])
- 适用场景: 短语查询、邻近查询、词性标注检索

**块索引（Blocked/Skip Index）**:
- 定义: 在 posting list 中设置跳跃指针（skip pointers）
- 与标准形式的区别: 增加层级结构加速列表合并
- 适用场景: 长 posting list 的布尔查询优化

**动态索引（Dynamic Index）**:
- 定义: 支持实时更新的索引结构
- 与标准形式的区别: 采用双缓冲区或日志结构合并（LSM）
- 适用场景: 实时搜索、流式数据处理

## 三、形式化证明

### 3.1 核心定理

**定理 1**: 对于包含 n 个文档、M 个唯一词项的集合，倒排索引的空间复杂度上界为 $O(\sum_{t \in T} |D_t|)$，其中 $D_t$ 为包含词项 t 的文档集合。

**证明**:
```
设 avg_tf 为平均每文档词项数，N 为总词项出现次数。

正排索引空间：
S_forward = Σ|d| = N × (avg_docID_size + avg_tf_size + overhead)

倒排索引空间：
S_inverted = M × dict_entry_size + Σ|D_t| × posting_size

由于 Σ|D_t| = Σ df(t) ≤ N，且通常 df(t) << N 对于大多数 t
根据 Zipf 定律：
- 排名 r 的词项的 collection frequency cf_r ∝ 1/r
- 只有少数高频词具有大的 posting list

因此 S_inverted << S_forward，通常压缩后为原数据的 20-30%。
```

**证明要点分析**:
1. **关键步骤1**: 利用 posting list 的有序性进行差分编码（D-Gap），将大数变为小数
2. **关键步骤2**: 应用 Zipf 定律说明大多数 posting list 很短，字典开销可控

**直觉理解**:
倒排索引利用"词项-文档"关系稀疏性（一个文档只包含少量词项，一个词项只出现在少量文档中），通过将 posting list 中的文档ID差分编码，实现高效压缩。

### 3.2 辅助引理

**引理 1**: 对于按文档ID排序的两个 posting list $L_1$ 和 $L_2$，求交集的时间复杂度为 $O(|L_1| + |L_2|)$。

*证明*: 
采用双指针合并算法：
```
p1 = p2 = 0
while p1 < |L1| and p2 < |L2|:
    if L1[p1] == L2[p2]:
        output L1[p1]
        p1++, p2++
    elif L1[p1] < L2[p2]:
        p1++
    else:
        p2++
```
每次迭代至少移动一个指针，总迭代次数不超过 $|L_1| + |L_2|$。∎

## 四、算法/方法详解

### 4.1 算法描述

**BSBI（Blocked Sort-Based Indexing）算法**:
```
算法: BSBI索引构建
输入: 文档集合 D，内存限制 M
输出: 倒排索引 I

1. 将文档集划分为大小为 M 的块 {B₁, B₂, ..., Bₖ}
2. for each block Bᵢ do
3.     解析 Bᵢ 中的文档，生成 (term, docID) 对
4.     在内存中按 (term, docID) 排序
5.     将排序后的结果写入磁盘文件 Fᵢ
6. end for
7. 合并所有 Fᵢ，相同 term 的 posting 聚合为 posting list
8. 写入最终的倒排索引文件
```

**流程图**:
```
文档集合 → [分块] → 块1, 块2, ..., 块k
              ↓
           [解析] → (term, docID) 对
              ↓
           [排序] → 按 term 然后 docID 排序
              ↓
           [写入] → 临时文件
              ↓
           [合并] → 多路归并
              ↓
           [输出] → 倒排索引
```

**SPIMI（Single-Pass In-Memory Indexing）算法**:
```
算法: SPIMI索引构建
输入: 文档集合 D，内存限制 M
输出: 倒排索引 I

1. output_files = []
2. while 还有未处理文档 do
3.     dictionary = 空哈希表
4.     while 内存未满 and 还有未处理文档 do
5.         doc = 读取下一个文档
6.         for each term in doc do
7.             if term 不在 dictionary then
8.                 dictionary[term] = 新 posting list
9.             end if
10.            将 docID 加入 dictionary[term] 的 posting list
11.        end for
12.    end while
13.    将 dictionary 按 term 排序后写入磁盘文件
14.    output_files.append(该文件)
15. end while
16. 合并所有 output_files
```

### 4.2 正确性分析

**BSBI不变式**: 在合并阶段开始时，每个临时文件中的 (term, docID) 对已经按字典序排列。

**证明**:
- **初始化**: 每个块内排序保证了单个文件的有序性
- **保持**: k路归并每次选取各文件当前最小的 term，保持输出有序
- **终止**: 当所有文件耗尽时，输出完整有序索引

**SPIMI不变式**: 处理过程中，每个 term 的 posting list 按 docID 递增顺序构建。

### 4.3 复杂度分析

**BSBI时间复杂度**:
- 最坏情况: $O(T \log T)$，T 为总词项数（主要是排序开销）
- 最好情况: $O(T)$（已排序输入）
- 平均情况: $O(T \log T)$

**BSBI空间复杂度**: $O(M)$，M 为内存限制

**复杂度证明**:
```
设 N 为总词项数，M 为内存容量，B = ⌈N/M⌉ 为块数

时间分析：
- 解析: O(N)
- 每块排序: B × O(M log M) = O(N log M)
- 归并: O(N log B) = O(N log(N/M))

总时间: O(N log N) （当 M 为常数时）

空间分析：
- 仅需常驻内存 M 的缓冲区
- I/O复杂度为 O(N/B) 块读写
```

**SPIMI优势**: 不需要全局排序，通过哈希直接构建 posting list，时间复杂度为 $O(N)$，但需要额外的排序步骤输出有序 posting list。

## 五、示例与实例

### 5.1 标准示例

**示例 1**: 简单倒排索引构建

**问题描述**:
给定三个文档：
- Doc1: "苹果 香蕉 苹果"
- Doc2: "香蕉 橙子"
- Doc3: "苹果 橙子 香蕉"

构建带词频的倒排索引。

**解决过程**:
1. **分词**: 已给出词项
2. **统计词频**:
   - "苹果": Doc1(2), Doc3(1)
   - "香蕉": Doc1(1), Doc2(1), Doc3(1)
   - "橙子": Doc2(1), Doc3(1)
3. **构建索引**:
   - 苹果 → [(1, 2), (3, 1)]
   - 香蕉 → [(1, 1), (2, 1), (3, 1)]
   - 橙子 → [(2, 1), (3, 1)]

**结果**: 查询"香蕉 AND 橙子" → 文档2和文档3

### 5.2 代码实现

**语言**: Python

```python
"""
倒排索引实现示例
支持布尔查询（AND, OR, NOT）
"""

from collections import defaultdict
from typing import List, Set, Dict, Tuple

class InvertedIndex:
    def __init__(self):
        # 词项 -> {文档ID: 词频}
        self.index: Dict[str, Dict[int, int]] = defaultdict(lambda: defaultdict(int))
        self.doc_count = 0
        self.doc_lengths: Dict[int, int] = {}
    
    def add_document(self, doc_id: int, tokens: List[str]):
        """添加文档到索引"""
        self.doc_count = max(self.doc_count, doc_id)
        self.doc_lengths[doc_id] = len(tokens)
        
        for token in tokens:
            self.index[token][doc_id] += 1
    
    def get_posting_list(self, term: str) -> List[Tuple[int, int]]:
        """获取词项的倒排列表 (docID, tf)"""
        if term not in self.index:
            return []
        return sorted(self.index[term].items())
    
    def boolean_and(self, terms: List[str]) -> Set[int]:
        """布尔与查询"""
        if not terms:
            return set()
        
        # 获取所有倒排列表
        lists = [set(self.index[term].keys()) for term in terms if term in self.index]
        
        if not lists:
            return set()
        
        # 按列表长度排序，优先处理短列表
        lists.sort(key=len)
        
        # 交集操作
        result = lists[0]
        for lst in lists[1:]:
            result &= lst
            if not result:  # 早期终止
                break
        
        return result
    
    def boolean_or(self, terms: List[str]) -> Set[int]:
        """布尔或查询"""
        result = set()
        for term in terms:
            if term in self.index:
                result.update(self.index[term].keys())
        return result
    
    def boolean_not(self, term: str, universe: Set[int] = None) -> Set[int]:
        """布尔非查询"""
        if universe is None:
            universe = set(range(1, self.doc_count + 1))
        
        if term not in self.index:
            return universe
        
        return universe - set(self.index[term].keys())
    
    def compute_tfidf(self, term: str, doc_id: int) -> float:
        """计算 TF-IDF 权重"""
        import math
        
        # 词频
        tf = self.index.get(term, {}).get(doc_id, 0)
        if tf == 0:
            return 0.0
        
        # 对数词频
        tf_weight = 1 + math.log10(tf)
        
        # 逆文档频率
        df = len(self.index.get(term, {}))
        idf = math.log10(self.doc_count / df) if df > 0 else 0
        
        return tf_weight * idf
    
    def compute_bm25(self, term: str, doc_id: int, k1: float = 1.2, b: float = 0.75) -> float:
        """计算 BM25 权重"""
        import math
        
        tf = self.index.get(term, {}).get(doc_id, 0)
        if tf == 0:
            return 0.0
        
        df = len(self.index.get(term, {}))
        idf = math.log((self.doc_count - df + 0.5) / (df + 0.5) + 1)
        
        # 平均文档长度
        avgdl = sum(self.doc_lengths.values()) / len(self.doc_lengths) if self.doc_lengths else 0
        dl = self.doc_lengths.get(doc_id, 0)
        
        # BM25 公式
        numerator = tf * (k1 + 1)
        denominator = tf + k1 * (1 - b + b * dl / avgdl)
        
        return idf * numerator / denominator
    
    def query_score(self, query_terms: List[str], doc_id: int, scoring: str = "tfidf") -> float:
        """计算查询与文档的相似度得分"""
        score = 0.0
        for term in query_terms:
            if scoring == "tfidf":
                score += self.compute_tfidf(term, doc_id)
            elif scoring == "bm25":
                score += self.compute_bm25(term, doc_id)
        return score


# 使用示例
if __name__ == "__main__":
    # 创建索引
    idx = InvertedIndex()
    
    # 添加文档
    documents = [
        (1, ["信息", "检索", "技术", "信息"]),
        (2, ["检索", "算法", "数据结构"]),
        (3, ["信息", "抽取", "技术"]),
        (4, ["机器学习", "算法", "技术"]),
    ]
    
    for doc_id, tokens in documents:
        idx.add_document(doc_id, tokens)
    
    # 布尔查询
    print("查询 '信息 AND 技术':", idx.boolean_and(["信息", "技术"]))
    print("查询 '检索 OR 算法':", idx.boolean_or(["检索", "算法"]))
    
    # TF-IDF 排序
    query = ["信息", "检索"]
    scores = [(doc_id, idx.query_score(query, doc_id, "tfidf")) for doc_id in range(1, 5)]
    scores.sort(key=lambda x: x[1], reverse=True)
    print("\nTF-IDF 排序结果:", scores)
    
    # BM25 排序
    scores = [(doc_id, idx.query_score(query, doc_id, "bm25")) for doc_id in range(1, 5)]
    scores.sort(key=lambda x: x[1], reverse=True)
    print("BM25 排序结果:", scores)
```

**代码说明**:
- **倒排索引结构**: 使用 `defaultdict` 嵌套实现 term → docID → tf 的映射
- **布尔查询**: 支持 AND（交集）、OR（并集）、NOT（补集）操作
- **TF-IDF/BM25**: 实现了两种经典排序算法
- **早期终止**: 在布尔 AND 查询中，当结果为空时提前退出

## 5.3 反例
### 5.3 反例

**常见错误**:

**错误1**: 未归一化词项导致索引膨胀
```python
# 错误示例
idx.add_document(1, ["Apple", "apple", "APPLE"])  # 被视为三个不同词项

# 正确做法
def normalize(token):
    return token.lower().strip()  # 统一小写

tokens = [normalize(t) for t in raw_tokens]
```
**错误原因**: 大小写敏感导致同一词项的变体被分散到不同 posting list
**正确做法**: 统一使用小写（或标准形态），考虑词干提取（stemming）

**错误2**: 忽视停用词导致查询效率低下
```python
# 问题
query = "the quick brown fox"
# "the" 可能出现在 90% 的文档中，处理成本极高

# 优化方案
STOPWORDS = {"the", "a", "an", "is", "are", ...}
def filter_stopwords(tokens):
    return [t for t in tokens if t not in STOPWORDS]
```

## 六、习题

### 6.1 理解题

1. **为什么倒排索引比正排索引更适合全文检索？** [难度⭐]
   
   *提示*: 从查询时间复杂度和空间局部性角度思考
   
   <details>
   <summary>解答</summary>
   
   正排索引（文档→词项）回答"文档包含哪些词"，倒排索引（词项→文档）回答"哪些文档包含该词"。
   
   全文检索的查询是"找包含词 X 的文档"，倒排索引可直接定位，时间复杂度 O(1) 查字典 + O(k) 遍历 posting list。
   
   若使用正排索引，需要扫描所有文档，时间复杂度 O(N × avg_doc_length)。
   
   此外，倒排索引可通过压缩 posting list 获得更优的空间效率。
   
   </details>

2. **解释 Zipf 定律对索引压缩的意义** [难度⭐⭐]
   
   <details>
   <summary>解答</summary>
   
   Zipf 定律：排名 r 的词项的出现频率 cf_r ∝ 1/r。
   
   这意味着：
   - 极少数高频词（排名靠前）占据了大部分文本
   - 绝大多数词是低频词，posting list 很短
   
   对压缩的意义：
   1. 短 posting list 使用变长编码更高效
   2. 高频词的 posting list 虽长，但差值编码后数值小，压缩率高
   3. 字典大小受限于词表，远小于文档集大小
   
   </details>

### 6.2 应用题

1. **设计一个高效的短语查询算法** [难度⭐⭐]
   
   给定查询"机器学习"，要求返回包含连续这两个词的文档。
   
   <details>
   <summary>解答</summary>
   
   算法步骤：
   1. 获取"机器"和"学习"的带位置信息的 posting list
      - "机器": [(D1, [3, 10]), (D3, [5]), ...]
      - "学习": [(D1, [4, 11]), (D3, [6]), ...]
   
   2. 对两个 posting list 求交集（文档ID层面）
      - 共同文档：D1, D3
   
   3. 对每个共同文档，检查位置差值：
      - D1: "机器"位置 [3, 10]，"学习"位置 [4, 11]
      - 检查：4-3=1 ✓，11-10=1 ✓
      - D1 包含短语
   
   4. 返回满足位置关系的文档
   
   优化：使用位置交集算法，时间复杂度 O(p + q)，p、q 为位置列表长度。
   
   </details>

2. **实现 Gamma 编码** [难度⭐⭐⭐]
   
   对 posting list [1, 3, 4, 8]（差值编码后）进行 Gamma 编码。
   
   <details>
   <summary>解答</summary>
   
   Gamma 编码步骤：
   1. 差值编码：1, 3-1=2, 4-3=1, 8-4=4 → [1, 2, 1, 4]
   
   2. Gamma 编码每个数 N：
      - 一元编码 floor(log₂N) 个 1，后接一个 0
      - 二进制编码 N - 2^floor(log₂N)，使用 floor(log₂N) 位
   
   编码过程：
   - 1: log₂1=0 → 一元部分：0，偏移部分：无 → `0`
   - 2: log₂2=1 → 一元部分：10，偏移：0 → `10 0` = `100`
   - 1: → `0`
   - 4: log₂4=2 → 一元部分：110，偏移：00 → `110 00` = `11000`
   
   结果：`0 100 0 11000` = `0100011000`（10位）
   
   对比：原始 4 个 int32 需 128 位，压缩比约 12.8:1。
   
   </details>

### 6.3 证明题

1. **证明 BM25 在文档无限长时的行为** [难度⭐⭐⭐]
   
   证明：当文档长度 dl → ∞ 时，BM25 的 TF 组件趋近于 k1 + 1。
   
   <details>
   <summary>解答</summary>
   
   BM25 公式：
   $$\text{score} = \text{IDF} \times \frac{\text{tf} \times (k_1 + 1)}{\text{tf} + k_1 \times (1 - b + b \times \frac{dl}{\text{avgdl}})}$$
   
   当 dl → ∞：
   - 假设 tf 与 dl 成正比（即词项均匀分布），tf = c × dl
   
   代入：
   $$\frac{c \cdot dl \cdot (k_1 + 1)}{c \cdot dl + k_1 \cdot b \cdot \frac{dl}{\text{avgdl}} + k_1(1-b)}$$
   
   分子分母同除以 dl：
   $$\frac{c(k_1+1)}{c + \frac{k_1 b}{\text{avgdl}} + \frac{k_1(1-b)}{dl}}$$
   
   当 dl → ∞：
   $$\to \frac{c(k_1+1)}{c + \frac{k_1 b}{\text{avgdl}}} < k_1 + 1$$
   
   这表明 BM25 通过文档长度归一化，避免了长文档的过度加分。
   
   </details>

### 7.2 实际案例

**案例**: Elasticsearch 在电商搜索中的应用

**背景**:
某电商平台日均 PV 1000万，商品库 500万 SKU，用户搜索查询 QPS 2000，要求 P99 响应时间 < 100ms。

**应用方式**:
1. **索引构建**: 使用 Logstash 同步 MySQL 商品数据到 ES
2. **分词策略**: 使用 IK 分词器，支持细粒度切分和同义词扩展
3. **查询优化**: 
   - 使用 filter context 缓存布尔过滤结果
   - 使用 function_score 实现个性化排序

**效果**:
- 平均查询延迟：15ms
- P99 延迟：45ms
- 搜索转化率提升 23%

### 7.3 跨领域联系

**与数据库的联系**:
关系型数据库的 B+ 树索引适合精确查询和范围查询，倒排索引适合多词项的集合查询。现代数据库（如 PostgreSQL）已内置全文检索功能，使用 GIN/GiST 索引实现倒排索引。

**与机器学习的联系**:
传统倒排索引基于词项匹配（lexical matching），神经网络检索（如 DPR）基于语义向量匹配。两者正在融合：稀疏-稠密混合检索（SPLADE, ColBERT）在保持可解释性的同时提升语义理解能力。

## 八、多维对比

### 8.1 方法对比

| 维度 | TF-IDF | BM25 | 语言模型 |
|------|--------|------|----------|
| 理论基础 | 向量空间模型 | 概率检索模型 | 概率语言模型 |
| TF 处理 | log(tf) | 饱和度函数 | 多项式/Dirichlet |
| 长度归一化 | 余弦归一化 | 参数化归一化 | 先验概率 |
| IDF | log(N/df) | log((N-df+0.5)/(df+0.5)) | 不直接使用 |
| 参数数量 | 0 | 2 (k1, b) | 1 (μ) |
| 效果 | 基线水平 | 通常最优 | 效果接近BM25 |

**压缩算法对比**:

| 算法 | 压缩率 | 解码速度 | 适用场景 |
|------|--------|----------|----------|
| Gamma | 高 | 慢 | 学术验证、小规模 |
| Delta | 很高 | 慢 | 差值变化大时 |
| VByte | 中等 | 很快 | 生产环境首选 |
| Simple9 | 中等 | 快 | 通用场景 |
| PForDelta | 高 | 很快 | 高吞吐量系统 |

### 8.2 决策建议

**何时选择 TF-IDF**:
- 教学或快速原型验证
- 文档长度差异不大的集合
- 需要简单、无参数调优的方案

**何时选择 BM25**:
- 生产环境搜索引擎
- 文档长度差异显著的集合（如网页搜索）
- 需要精细控制长度归一化（调整 b 参数）

**决策流程图**:
```
开始
  │
  ▼
文档长度差异大？
  │
  ├─是 → 使用 BM25
  │       │
  │       ▼
  │   长文档惩罚严重？
  │       │
  │       ├─是 → 降低 b 参数（0.3-0.5）
  │       └─否 → 使用标准 b=0.75
  │
  └─否 → 文档数量大？
          │
          ├─是 → 使用 BM25（扩展性更好）
          └─否 → TF-IDF 足够
```
