## λ演算类型系统层级

| 系统 | 名称 | 特性 | 停机性 | 表达能力 | 对应语言 |
|------|------|------|--------|---------|---------|
| **λ→** | 简单类型λ演算 | 基础函数类型 | ✓ | 弱 | 简单函数式 |
| **λ2** | System F | 参数多态 | ✓ | 强 | Haskell核心 |
| **λω** | System Fω | 类型构造器多态 | ✓ | 很强 | Haskell+类型类 |
| **λP** | LF | 值依赖类型 | ✓ | 很强 | Twelf |
| **λC** | 演算构造 | 依值类型完整 | ✓ | 图灵完备 | Coq, Agda |
| **λ∞** | 无类型λ演算 | 无类型 | ✗ | 图灵完备 | Lisp, Scheme |

## 类型系统设计权衡

| 设计目标 | 类型系统选择 | 优点 | 缺点 | 典型应用 |
|----------|-------------|------|------|---------|
| **极致性能** | 静态+单态化 | 零开销抽象 | 代码膨胀 | 系统编程(Rust, C++) |
| **快速开发** | 动态+鸭子类型 | 灵活快速 | 运行时错误 | 脚本、原型(Python) |
| **大规模维护** | 静态+强类型+泛型 | 重构安全 | 前期成本 | 企业应用(Java, C#) |
| **形式化验证** | 依值类型 | 数学级正确性 | 开发极慢 | 关键系统(Coq, F*) |
| **函数式编程** | 代数数据类型+模式匹配 | 表达力强 | 学习曲线 | 数据处理(Haskell) |
| **渐进迁移** | 可选类型+ gradual typing | 渐进式升级 | 类型漏洞 | 大型旧项目(TypeScript) |

## 子类型关系对比

| 语言 | 子类型规则 | 型变支持 | 边界类型 | 检查时机 |
|------|-----------|---------|---------|---------|
| **Java** | 名义子类型 | 协变(数组)、不变(泛型)、使用点型变 | 上/下界 | 编译+运行时 |
| **C++** | 名义子类型 | 模板协变/逆变、隐式转换 | 概念(C++20) | 编译 |
| **Scala** | 名义+结构子类型 | 声明点型变 | 上/下界 | 编译 |
| **TypeScript** | 结构子类型 | 协变/逆变 | 条件类型 | 编译(擦除) |
| **Rust** | 名义子类型(trait) | 无子类型(只有特化) | Trait bound | 编译 |
| **Go** | 结构子类型(接口) | 无 | 接口 | 编译 |

## 类型推断能力对比

| 算法/系统 | 推断范围 | 多态支持 | 复杂度 | 全局推断 | 应用 |

| **约束基础推断** | 全局 | 高阶多态 | 高 | ✓ | HML |

## 类型系统选择决策

```
项目需求?
    │
    ├── 极致性能+系统级?
    │       │
    │       ├── 需要内存安全? → Rust
    │       │
    │       └── 传统选择? → C++
    │
    ├── 大型企业应用?
    │       │
    │       ├── JVM生态? → Java/Kotlin/Scala
    │       │
    │       └── .NET生态? → C#/F#
    │
    ├── 快速开发/数据科学?
    │       │
    │       ├── 类型安全重要? → Python+Type Hint
    │       │
    │       └── 函数式优先? → Haskell
    │
    ├── Web前端?
    │       └── TypeScript (行业标准)
    │
    ├── 函数式编程?
    │       │
    │       ├── 纯函数式? → Haskell
    │       │
    │       └── 实用函数式? → Scala, F#
    │
    ├── 形式化验证?
    │       │
    │       ├── 定理证明? → Coq, Agda
    │       │
    │       └── 验证工具? → F*, Isabelle
    │
    └── 云原生/并发?
            └── Go (简单高效), Erlang/Elixir (容错)
```

---

*注: 类型系统的选择取决于项目需求、团队熟悉度和生态系统。现代趋势是静态类型+强类型+更好推断+gradual typing。*
