## 第一部分：为什么需要复杂度分析 (5分钟)

### 场景引入 (画面：分屏对比)

**左屏**：一个 $O(n)$ 算法处理 1000 万数据，耗时 0.1 秒
**右屏**：一个 $O(n^2)$ 算法处理 1000 万数据，预计耗时 11.5 天

### 讲师
> "假设你开发了两个排序算法。在小数据量时，它们都很快。
> 但当数据量达到 1000 万时，一个瞬间完成，另一个...可能需要等到下周。
>
> 这就是复杂度分析的威力——它让我们在写第一行代码前，就能预测算法的性能边界。"

### 关键概念

```
┌─────────────────────────────────────────────────────┐
│           性能评估的两种方法                          │
├─────────────────────────────────────────────────────┤
│  事后统计 (Profiling)    │   事前分析 (Analysis)     │
├─────────────────────────────────────────────────────┤
│  • 实际运行测量          │   • 数学推导预测          │
│  • 依赖硬件环境          │   • 独立于具体机器        │
│  • 需要实现后才能测试    │   • 设计阶段即可评估      │
│  • 结果精确但局限        │   • 结果抽象但普适        │
└─────────────────────────────────────────────────────┘
```

### 讲师强调
> "复杂度分析是算法的'体检报告'，它告诉我们算法随输入规模增长时的行为特征。"

## 第二部分：大 O 表示法 (8分钟)

### 2.1 直观理解 (画面：函数增长曲线动画)

展示不同复杂度函数的图像：
- $O(1)$ — 水平线
- $O(\log n)$ — 平缓上升
- $O(n)$ — 45度斜线
- $O(n \log n)$ — 略陡于线性
- $O(n^2)$ — 抛物线
- $O(2^n)$ — 近乎垂直

### 讲师
> "大 O 表示法描述的是算法运行时间的'上界'。
> 你可以把它想象成一个'最坏情况保证'——无论输入如何，算法的性能都不会比这个更差。"

### 2.2 数学定义

**屏幕显示**：

$$O(g(n)) = \{f(n) \mid \exists c > 0, n_0 > 0, \forall n \geq n_0: 0 \leq f(n) \leq c \cdot g(n)\}$$

### 讲师解释
> "这个定义看起来很复杂，但其实很简单：
> - 存在常数 $c$ 和 $n_0$
> - 当 $n$ 足够大时（超过 $n_0$）
> - $f(n)$ 不会超过 $c \cdot g(n)$
>
> 简单说，就是当数据量很大时，$g(n)$ 是 $f(n)$ 的'上限'。"

### 2.3 计算示例

**画面**：代码高亮 + 步骤标注

```rust
fn find_max(arr: &[i32]) -> Option<i32> {
    if arr.is_empty() {
        return None;           // O(1)
    }
    
    let mut max = arr[0];      // O(1)
    
    for i in 1..arr.len() {    // 循环 n-1 次
        if arr[i] > max {      // O(1) 每次
            max = arr[i];      // O(1) 最坏情况
        }
    }
    
    Some(max)                  // O(1)
}
```

**计算过程**：
```
T(n) = O(1) + O(1) + (n-1) × [O(1) + O(1)] + O(1)
     = O(1) + O(n)
     = O(n)                           ← 保留最高阶项
```

### 讲师
> "分析代码时，我们：
> 1. 识别基本操作（赋值、比较、算术运算）
> 2. 统计每个基本操作的执行次数
> 3. 用 $O$ 表示法概括
> 4. 忽略低阶项和常数系数"

## 第三部分：常见复杂度类别 (8分钟)

### 3.1 复杂度速查表

| 复杂度 | 名称 | 示例 | 规模限制 |
|--------|------|------|----------|
| $O(1)$ | 常数时间 | 数组访问、哈希表查找 | 无限制 |
| $O(\log n)$ | 对数时间 | 二分搜索、平衡树操作 | $n \approx 10^{18}$ |
| $O(n)$ | 线性时间 | 遍历数组、线性搜索 | $n \approx 10^8$ |
| $O(n \log n)$ | 线性对数 | 快速排序、归并排序 | $n \approx 10^6$ |
| $O(n^2)$ | 平方时间 | 冒泡排序、简单矩阵运算 | $n \approx 10^4$ |
| $O(n^3)$ | 立方时间 | Floyd 最短路径（朴素） | $n \approx 500$ |
| $O(2^n)$ | 指数时间 | 子集枚举、旅行商问题（暴力） | $n \approx 25$ |
| $O(n!)$ | 阶乘时间 | 全排列枚举 | $n \approx 11$ |

### 3.2 实际意义演示 (画面：可视化对比)

假设每秒执行 1 亿次操作：

```
n = 1000 时：
┌─────────────────┬──────────────┐
│ O(log n)        │ 0.01 微秒    │
│ O(n)            │ 10 微秒      │
│ O(n log n)      │ 100 微秒     │
│ O(n²)           │ 10 毫秒      │
│ O(n³)           │ 10 秒        │
│ O(2ⁿ)           │ 宇宙年龄×100 │
└─────────────────┴──────────────┘
```

### 讲师
> "注意这个'宇宙年龄'——这不是夸张。
> $2^{1000}$ 是一个 302 位的数字，远超可观测宇宙中的原子总数。"

### 3.3 空间复杂度

**画面**：内存使用示意图

```rust
// 空间复杂度 O(1)
fn sum_iterative(arr: &[i32]) -> i32 {
    let mut sum = 0;                    // 1 个变量
    for &x in arr {
        sum += x;
    }
    sum
}

// 空间复杂度 O(n)
fn sum_recursive(arr: &[i32]) -> i32 {
    if arr.is_empty() {
        0
    } else {
        arr[0] + sum_recursive(&arr[1..])  // 递归栈深度 n
    }
}
```

### 讲师
> "空间复杂度的分析与时间复杂度类似，但要注意：
> - 输入空间通常不计入（除非算法修改输入）
> - 递归调用需要计算栈空间
> - 动态分配的内存要计入"

## 第四部分：分析技巧与陷阱 (5分钟)

### 4.1 分析法则

**画面**：公式卡片

```
1. 加法法则：T(n) = T₁(n) + T₂(n) = O(max(f(n), g(n)))
   
2. 乘法法则：T(n) = T₁(n) × T₂(n) = O(f(n) × g(n))

3. 忽略常数：O(2n) = O(n)

4. 忽略低阶项：O(n² + n) = O(n²)
```

### 4.2 常见陷阱

| 陷阱 | 示例 | 正确分析 |
|------|------|----------|
| 忽略隐藏循环 | `s.contains(x)` 在字符串中 | $O(n)$，不是 $O(1)$ |
| 误判递归 | 斐波那契递归 | $O(2^n)$，不是 $O(n)$ |
| 最优/最坏混淆 | 快速排序 | 最优 $O(n \log n)$，最坏 $O(n^2)$ |
| 均摊分析遗漏 | 动态数组扩容 | 单次 $O(n)$，均摊 $O(1)$ |

### 4.3 实战技巧

> **讲师**："当你不确定时，问自己三个问题：
> 1. 最坏情况下会发生什么？
> 2. 循环的迭代次数与什么有关？
> 3. 递归调用的深度是多少？"

> - 正确性证明
> - 应用场景约束

### 规范模板示例

```yaml
算法规范:
  名称: BinarySearch
  时间复杂度:
    最优: O(1)      # 目标在正中
    平均: O(log n)  # 随机分布
    最坏: O(log n)  # 目标在边界或不存在
  空间复杂度:
    迭代版: O(1)
    递归版: O(log n)  # 调用栈
  前置条件: 输入数组必须有序
  后置条件: 返回目标索引或 None
```

## 总结与作业 (2分钟)

### 核心要点回顾

1. **大 O 表示法**：描述算法随输入规模增长的上界
2. **常见复杂度**：常数 < 对数 < 线性 < 线性对数 < 平方 < 立方 < 指数 < 阶乘
3. **分析方法**：识别基本操作 → 统计执行次数 → 取最高阶项
4. **实践原则**：写代码前先分析，选择适合数据规模的算法

### 课后作业

1. **分析练习**：分析冒泡排序和插入排序的时间复杂度
2. **对比实验**：用代码验证不同复杂度算法的实际运行时间差异
3. **规范撰写**：为二分搜索撰写完整的时间/空间复杂度规范
