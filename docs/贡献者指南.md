# 贡献者指南

欢迎来到【算法规范与模型设计知识体系】项目！

## 关于项目

本项目是一个开源的算法理论文档项目，目标是建立系统、严谨、国际化的算法知识体系。项目包含200+文档，涵盖基础理论、递归理论、形式化证明、算法复杂度、类型理论、逻辑系统、计算模型等多个领域。

### 项目特点

- **规范导向**：以算法规范设计为核心，建立标准化设计框架
- **形式化方法**：采用严格的数学形式化表示
- **多表征方式**：规范定义、模型设计、数学符号、示例性代码片段
- **双语体系**：中英文双语呈现，便于国际化传播
- **开源协作**：欢迎全球贡献者参与建设

### 贡献流程

#### 第一步：选择任务
查看 [GitHub Issues](../../issues) 中的任务列表，或浏览文档发现需要改进的地方。

**任务标签说明：**
- `good first issue`：适合新手的简单任务
- `help wanted`：需要帮助的任务
- `bug`：文档中的错误
- `content-request`：内容补充请求
- `improvement`：改进建议

#### 第二步：认领任务
在Issue下评论：
```
我想认领这个任务，预计X天完成
```

如果发现了新问题，请先创建Issue描述问题。

#### 第三步：开展工作

1. **Fork 仓库**
   ```bash
   # 点击右上角的 Fork 按钮
   ```

2. **创建分支**
   ```bash
   git checkout -b feature/[任务描述]
   # 例如：git checkout -b feature/fix-typo-in-recursion-theory
   ```

3. **提交更改**
   ```bash
   git add .
   git commit -m "类型: 简短描述"
   # 例如：git commit -m "fix: 修正递归函数定义中的拼写错误"
   ```

#### 第四步：提交PR

1. 推送到你的Fork
   ```bash
   git push origin feature/[任务描述]
   ```

2. 在GitHub上创建Pull Request
   - 使用PR模板填写信息
   - 关联相关Issue（使用 `Fixes #123` 格式）
   - 等待审查

#### 第五步：合并与致谢

- 解决审查意见
- PR合并后，你的名字将被添加到[贡献者名单](#贡献者名单)

## 文档贡献规范

### 文档结构

每个文档必须包含以下部分：

```markdown
# 文档标题

## 文档头
- **版本**: vx.x.x
- **日期**: YYYY-MM-DD
- **状态**: [草案/审查中/已发布]
- **作者**: [作者名]

## 学习目标
- 学习目标1
- 学习目标2

## 正文内容
### 定义
[严格的形式化定义]

### 定理
[定理陈述]

#### 证明
[形式化证明过程]

### 示例
[代码示例]

## 参考文献
1. [作者]. [标题]. [出版信息].

## 知识导航
- **前置知识**: [链接]
- **后续主题**: [链接]
- **相关主题**: [链接]
```

### 写作风格

#### 语言规范
- **双语呈现**：标题、重要概念使用中英文双语
- **术语统一**：遵循项目术语表，首次出现时使用双语标注
- **简洁清晰**：避免冗长，追求精确表达

#### 数学符号
- 遵循 ISO 80000-2 国际标准
- 使用 LaTeX 格式书写数学公式
- 符号使用前需明确定义

#### 引用规范
- 使用 ACM 引用格式
- 所有定义必须有来源引用
- 所有定理证明需引用原始文献

#### 代码示例
- 使用 Rust 语言
- 代码需包含完整类型签名
- 添加必要的注释说明

### 质量要求

| 检查项 | 要求 | 检查方式 |
|--------|------|----------|
| 文档结构 | 符合模板 | 手动检查 |
| 数学符号 | ISO 80000-2 | 手动检查 |
| 引用格式 | ACM格式 | 手动检查 |
| 双语标注 | 标题+重要概念 | 手动检查 |
| 交叉引用 | 链接正确 | 自动化检查 |
| 拼写错误 | 无 | 自动化检查 |

## 代码贡献规范

### 代码标准

- **语言**: Rust
- **代码风格**: 遵循 `rustfmt` 标准格式
- **文档注释**: 使用 `///` 格式编写文档
- **测试覆盖**: 单元测试覆盖率 > 80%

### 代码示例模板

```rust
/// 函数功能描述
/// 
/// # 参数
/// - `param`: 参数说明
/// 
/// # 返回值
/// 返回值说明
/// 
/// # 示例
/// ```
/// let result = function_name(arg);
/// ```
/// 
/// # 复杂度
/// - 时间复杂度: O(?)
/// - 空间复杂度: O(?)
pub fn function_name(param: Type) -> ReturnType {
    // 实现
}

#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_function_name() {
        assert_eq!(function_name(test_input), expected_output);
    }
}
```

## 行为准则

### 我们承诺

- **尊重所有参与者**：无论背景、经验水平如何
- **保持专业态度**：专注于技术和内容
- **接受建设性批评**：开放心态，持续改进
- **关注技术本身**：避免人身攻击和歧视性言论
- **保护知识产权**：遵守开源许可证规定

### 不可接受的行为

- 歧视性言论或骚扰
- 人身攻击或侮辱
- 恶意破坏或刷屏
- 未经授权的商业推广

### 常见问题

**Q: 我是新手，可以贡献吗？**
A: 当然可以！查看带有 `good first issue` 标签的任务，从简单的拼写修正或格式改进开始。

**Q: 如何报告问题？**
A: 使用对应的Issue模板创建问题，提供尽可能详细的信息。

**Q: 贡献需要签署CLA吗？**
A: 不需要。但提交PR即表示你同意将贡献内容以Apache 2.0许可证授权。

**Q: 我的贡献会被如何认可？**
A: 所有贡献者都会被列入贡献者名单，活跃贡献者会被提名为核心贡献者。
